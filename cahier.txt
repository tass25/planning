1. Context and Problem Statement
1.1 Industrial Context
Organizations that have operated for decades on SAS (Statistical Analysis System) environments face mounting pressure to migrate their analytical infrastructure to modern, open-source Python stacks. The primary drivers are:

Licensing cost: SAS enterprise licenses average $8,000–$100,000+ per seat annually. Python is free and open-source.
Ecosystem velocity: Python's data science ecosystem (pandas, scikit-learn, PyTorch, statsmodels) advances at a pace SAS cannot match.
Talent availability: The global pool of Python developers vastly exceeds that of SAS programmers. New graduates enter the market with Python skills, not SAS.
Infrastructure modernization: Cloud-native architectures (AWS, GCP, Azure) are built around Python, not SAS.

Manual migration is not a viable solution at scale. A single enterprise SAS codebase can contain thousands of scripts, hundreds of thousands of lines of code, spanning ETL pipelines, statistical analyses, regulatory reporting (particularly in pharma and finance), and complex macro libraries with deeply nested inter-file dependencies. Manual translation by a developer familiar with both languages costs approximately 3–8 hours per 100 lines of SAS code. For a 500,000-line codebase, this means 15,000–40,000 hours of developer time — effectively years of effort.
1.2 Problem Statement
The core technical challenge is not syntactic translation — a rule-based regex system can handle the easy cases. The challenge is semantic equivalence under complexity:

SAS has unique date arithmetic (epoch: January 1, 1960 vs Python's 1970) that produces silent numerical errors when naively translated.
SAS's MERGE statement has specific sorted-join semantics fundamentally different from pd.merge().
The RETAIN statement creates persistent row-to-row state in a DATA step — a concept with no direct Python equivalent.
FIRST. and LAST. automatic variables depend on BY-group processing semantics not present in pandas.
Macro %INCLUDE chains can create circular dependencies spanning dozens of files, requiring dependency graph analysis before any translation can begin.
Missing value semantics differ: SAS numeric missing (.) behaves as negative infinity in comparisons, while Python's NaN returns False in all comparisons.

These failure modes compound: an LLM that doesn't know about the 1960 epoch will silently produce wrong date calculations in every converted script. A system with no dependency graph will translate blocks in the wrong order, producing scripts that reference DataFrames that don't exist yet.
1.3 Solution Approach
This project delivers an AI-powered, agentic SAS-to-Python conversion accelerator that combines:

Semantic partitioning — decompose SAS scripts into logical, typed blocks with full dependency metadata, using RAPTOR-inspired hierarchical clustering (Sarthi et al., ICLR 2024) rather than flat keyword splitting.
Knowledge-base-augmented translation — retrieve verified SAS→Python/PySpark example pairs before translating each block, providing the LLM with domain-correct few-shot context, particularly for known failure modes.
Calibrated quality assurance — every translated block is cross-verified by a second LLM call with structured JSON output, and quality metrics are tracked over time to detect and correct drift.
Structured script reconstruction — translated blocks are merged in correct execution order with consolidated imports, resolved variable names, and syntax validation before output.
Continuous learning — corrections feed back into the knowledge base automatically, making the system improve with every run.


2. Objectives
2.1 Primary Objectives
O1 — End-to-end conversion pipeline: The system must accept any valid SAS file as input and produce an equivalent, syntactically valid Python (or PySpark) script as output, without requiring manual intervention for standard SAS constructs (DATA steps, PROC blocks, PROC SQL, standard macros).
O2 — Semantic correctness: The produced Python code must be functionally equivalent to the SAS source, with explicit handling of the six documented failure modes (date arithmetic, MERGE semantics, RETAIN, FIRST./LAST., missing value comparisons, PROC MEANS OUTPUT OUT=).
O3 — Coverage breadth: The system must handle all major SAS usage categories: ETL pipelines, statistical analyses, SQL operations, reporting procedures, macro programming, and file I/O — not just simple DATA steps.
O4 — Measurable quality: Every component must have a quantitative quality target with an automated evaluation suite. Accuracy claims must be verifiable by the jury.
O5 — Research contribution: The partition layer must constitute an original academic contribution by adapting RAPTOR (Sarthi et al., ICLR 2024) to SAS code block grouping, validated by an ablation study comparing RAPTOR retrieval against flat-index retrieval.
2.2 Secondary Objectives
O6 — Fault tolerance: The pipeline must support checkpoint-and-resume so that a crash during processing of a large SAS project does not require starting over.
O7 — Continuous learning: The system must improve over time by ingesting corrections (human or automated) back into the knowledge base.
O8 — 100% open-source stack: No proprietary dependencies (no OpenAI API, no Neo4j GPL, no SAS runtime). Every component must be deployable on a student machine without cost.

4. Component Specifications
4.1 L2-A — Entry & Scan
FileAnalysisAgent
Discovers all .sas files under a project root using pathlib.Path.rglob("*.sas"). For each file: reads raw bytes, detects encoding with chardet, decodes content, computes SHA-256 content hash, counts lines, and runs a Lark LALR pre-validation pass to detect structural errors (unclosed %MACRO definitions, truncated PROC blocks, encoding corruption artifacts) before any streaming begins. Files failing pre-validation are logged to the registry with status=ERROR and skipped — preventing garbage from propagating through seven downstream layers.
Output: FileMetadata — {file_id, file_path, encoding, content_hash, file_size_bytes, line_count, lark_valid, lark_errors, created_at}
CrossFileDependencyResolver (new component, not in naive designs)
Before streaming begins, performs a project-wide regex scan for inter-file references: %INCLUDE 'path', LIBNAME name 'path', and macro variable includes %INCLUDE &var. For each reference, attempts to resolve the target path relative to the file's directory and the project root. Resolved references are written to cross_file_deps (SQLite table in file_registry.db). Unresolved references (macro variable paths, external network paths) are logged with resolved=False. This data is consumed by IndexAgent in L2-E to build accurate DEPENDS_ON edges in the NetworkX graph without guessing.
RegistryWriterAgent
Idempotent write of FileMetadata to SQLite (dev) / PostgreSQL (prod) using SQLAlchemy. Deduplication on content_hash — if the same file is scanned twice (e.g., after a crash restart), the second scan produces no new rows. Uses INSERT OR IGNORE semantics.
4.2 L2-B — Streaming Core
StreamAgent
Asynchronous file reader using aiofiles. Opens each SAS file with encoding from FileRegistry, reads in an 8 KB internal buffer, and yields LineChunk objects one at a time without loading the full file into memory. Supports files of arbitrary size — tested up to 500 MB (2+ million lines) without memory overflow. Implements producer side of the backpressure queue.
StateAgent
Single-pass finite state machine running per LineChunk, updating ParsingState in under 0.5ms per line (no LLM, no I/O — pure Python logic):

block_type transitions: IDLE → DATA_STEP → PROC_BLOCK → MACRO_DEF → SQL_BLOCK → CONDITIONAL_BLOCK → LOOP_BLOCK → GLOBAL_STATEMENT → INCLUDE_REFERENCE
nesting_depth: increments on %DO/%IF, decrements on %END, floor at 0
macro_stack: push on %MACRO name, pop on %MEND — tracks macro call depth
variable_scope: live dictionary of variable assignments seen so far
active_dependencies: live set of LIBNAME/%INCLUDE references encountered

BackpressureQueue
asyncio.Queue(maxsize=200). Producer (StreamAgent) blocks on await queue.put() when full; consumer (BoundaryDetectorAgent) blocks on await queue.get() when empty. This bounds memory consumption regardless of file size — without it, a 500,000-line file produces 500,000 pending LineChunk objects before RAPTOR can process the first batch.
For files > 100 MB: maxsize drops to 50. For files > 500 MB: sequential processing (one file at a time, 5,000-line segments, Parquet IR store).
4.3 L2-C — RAPTOR Chunking
This is the system's primary academic contribution. It adapts RAPTOR (Sarthi et al., ICLR 2024) from long-document question-answering to SAS code block grouping, enabling downstream RAG layers to retrieve at exactly the right abstraction level: leaf for simple blocks, cluster for semantically related procedure families, root for file-level planning context.
BoundaryDetectorAgent
Lark LALR grammar (sas.lark) detects canonical SAS block boundaries from the ParsingState stream. Handles all 9 canonical partition types:
Partition TypeDetection Triggertest_coverage_typeDATA_STEPLine starts with DATA fullPROC_BLOCKLine starts with PROC fullMACRO_DEFINITIONLine starts with %MACRO fullMACRO_INVOCATION%macroname() call patternfullSQL_BLOCKPROC SQL specificallyfullCONDITIONAL_BLOCK%IF/%THEN/%ELSE outside PROCstructural_onlyLOOP_BLOCK%DO/%END iterativestructural_onlyGLOBAL_STATEMENTOPTIONS, LIBNAME, FILENAME, TITLEstructural_onlyINCLUDE_REFERENCE%INCLUDE statementstructural_only
Blocks open for >200 lines with no closing keyword are flagged AMBIGUOUS and routed to the LLM resolver.
Boundary detection is intentionally cheap: no embedding, no LLM, no I/O beyond pattern matching. This handles approximately 80% of all blocks.
LLM Boundary Resolver (Llama 3.1 8B via Ollama)
For AMBIGUOUS blocks only (~20% of total). Uses the instructor library to enforce a typed BlockBoundaryEvent Pydantic output — no JSON parsing errors. Before calling Ollama, queries KBQueryClient.query_similar(raw_lines, block_type, k=3) to retrieve relevant KB examples for few-shot context. Tiktoken guard: prompts exceeding 6,000 tokens are truncated (raw code section first, instructions preserved). Retry: 3× exponential backoff (base 1s). If all retries fail: emit rule-based boundary with boundary_method="lark_fallback" and log to DuckDB audit.
PartitionBuilderAgent
Assembles BlockBoundaryEvent + ParsingState metadata into flat PartitionIR objects. Sets partition_type from block type, content_hash from SHA-256 of raw_code, test_coverage_type from partition type (full vs structural_only), carries trace_id from orchestrator. RAPTOR fields (raptor_leaf_id, raptor_cluster_id, raptor_root_id) are empty at this stage — filled by RAPTORPartitionAgent.
NomicEmbedder
Embeds each PartitionIR.raw_code block into a 768-dimensional vector using Nomic Embed v1.5 (nomic-ai/nomic-embed-text-v1.5, Apache 2.0, fully local via sentence-transformers). Batch size 32 to avoid CUDA OOM. Sets PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:512. Caches in memory as dict[partition_id, np.ndarray]. For files >100 MB, only HIGH-risk blocks are embedded (reduces memory pressure from GMM on large matrices).
GMMClusterer
Following Sarthi et al. (2024) §3.1: fits sklearn.mixture.GaussianMixture on L2-normalized leaf embeddings. k selected by BIC minimization over k ∈ [max(2,1), min(⌊√N⌋, 20)] — the √N formula is directly from the paper. Similarity threshold τ=0.72: blocks whose maximum cluster membership probability falls below τ are marked cluster_label=-1 (outliers) and remain as standalone leaves. min_cluster_size=2: singleton clusters are merged with their cosine-nearest neighbour. Fallback: if GMM fitting fails after 3 attempts (numerical divergence, n_blocks < 4), all blocks get cluster_label=0 (flat partitioning).
ClusterSummarizer
Generates a natural-language summary of each cluster using Llama 3.1 70B via Groq API. Prompt: "Given these SAS code blocks, write a concise technical summary (max 150 words) describing their collective purpose, the data they process, and how they relate to each other." Hash-cached: if the SHA-256 of the cluster's sorted embedding matrix matches a previous call, the cached summary is returned without an LLM call (critical for enterprise codebases where the same macro family appears repeatedly). Sets summary_tier on RAPTORNode: "groq" | "ollama_fallback" | "heuristic_fallback". Groq 429 → 3× exp backoff (base 2s) → Ollama 70B local → heuristic summary (partition_type count string).
RAPTORTreeBuilder
Recursive tree construction algorithm (Sarthi et al. §3.2):

Leaf nodes: each PartitionIR is one RAPTORNode at level=0, embedding already computed.
GMM clustering on leaf embeddings → cluster assignments.
For each cluster: call ClusterSummarizer → summary string.
Embed each cluster summary with Nomic Embed → cluster-level 768-dim vector.
Create RAPTORNode at level=1 (L1 cluster), child_ids = member partition_ids.
Convergence check: stop if k=1 OR current_depth >= max_depth OR |BIC_t − BIC_{t-1}| < 0.01.
If not converged: recurse — treat L1 cluster nodes as new leaves, repeat from step 2.
Final root node: single RAPTORNode at level=3, full-file summary.

Dynamic depth: max_depth=3 for standard files; max_depth=5 when macro_density > 0.4 (macro-heavy enterprise files benefit from deeper clustering). All nodes stored in LanceDB raptor_nodes table.

FLAT_PARTITION: score < 0.35, no macros, cross_file_deps < 3
MACRO_AWARE: has_macros=True OR score 0.35–0.68
DEPENDENCY_PRESERVING: cross_file_deps > 3 OR score 0.50–0.68
STRUCTURAL_GROUPING: score > 0.68 OR has_nested_sql=True

The model is trained on 500 manually-labeled examples (derived from the 50-file gold standard corpus — each annotated block becomes one training row). Platt scaling is critical for calibrated probability output: a raw logistic regression confidence of 0.73 is not reliable; post-hoc Platt scaling corrects this. ECE (Expected Calibration Error) is tracked every training run in DuckDB calibration_log. Target ECE < 0.08.
4.5 L2-E — Persistence & Index
PersistenceAgent
Writes PartitionIR[] to:

SQLite (dev, < 10,000 blocks): SQLAlchemy ORM, INSERT OR IGNORE dedup on content_hash
PostgreSQL (prod): same ORM, connection pool
Parquet (batches ≥ 10,000 blocks): pyarrow serialization, written to data/partitions/file_{file_id}.parquet

RAPTORWriter
Inserts all RAPTORNode objects (leaf + cluster + root) into LanceDB raptor_nodes table with cosine IVF index (num_partitions=32). Updates PartitionIR.raptor_leaf_id, raptor_cluster_id, raptor_root_id back to SQLite/Postgres.
IndexAgent
Builds the dependency graph in three stages:
Stage 1 — NetworkX DAG construction: For each PartitionIR, scans dependency_refs and cross-references against cross_file_deps (pre-resolved by CrossFileDependencyResolver in L2-A). Creates DEPENDS_ON edges for dataset references, LIBNAME refs, and resolved %INCLUDE targets. Creates MACRO_CALLS edges from macro_scope metadata.
Stage 2 — SCC Condensation: nx.strongly_connected_components(dag) finds all circular dependency cycles (A → B → C → A). SCCs with size > 1 are collapsed into super-nodes via nx.condensation(dag), guaranteeing an acyclic graph for downstream traversal queries. Each partition in a cycle receives scc_id — partitions sharing an scc_id must be translated together in L3.
Stage 3 — Dynamic Hop Cap: nx.dag_longest_path_length(condensed_dag) computes the longest dependency chain in the project. Result capped at 10, stored in config/project_config.yaml. All graph traversal queries use this value via nx.ancestors() bounded by max_hop.
RedisCheckpoint
Writes JSON-serialized PartitionIR[] to Redis every 50 processed blocks. Key format: partition:{file_id}:checkpoint:{block_num}, TTL 24 hours. On orchestrator startup: scans Redis for existing checkpoints for the current run — if found, resumes from the highest block number (prevents re-processing already-completed work after a crash). If Redis is unavailable: logs a warning and continues without checkpointing (degraded mode — full restart required on crash, but no data corruption).
4.6 Layer 3 — Translation
TranslationAgent
Per-block SAS→Python/PySpark conversion with four guarantees: (1) KB-augmented few-shot context for every LLM call, (2) failure-mode-aware prompting for the 6 known problem patterns, (3) cross-verification by a second LLM call for MODERATE/HIGH blocks, (4) output-target routing (pandas-based Python or PySpark) based on a target_runtime flag set by StrategyAgent (default: python; set to pyspark when the source SAS file references distributed data constructs or explicit user configuration).
Failure mode detection (rule-based, no LLM):

RETAIN: "RETAIN " in raw_code.upper()
FIRST_LAST: "FIRST." or "LAST." in code
DATE_ARITHMETIC: MDY( or TODAY() or INTNX( or INTCK( in code
MERGE_SEMANTICS: "MERGE " and "BY " both present
MISSING_VALUE_COMPARISON: comparison operators present AND has_macros=True
PROC_MEANS_OUTPUT: "OUTPUT OUT=" in code

KB retrieval: KBQueryClient.query_similar(raw_code, partition_type, k=5, failure_mode=detected_fm, target_runtime=target_runtime). Returns up to 5 verified SAS→Python/PySpark pairs from LanceDB, filtered by partition_type OR failure_mode AND target_runtime. These are injected into the translation prompt as few-shot examples.
LLM routing:

LOW → Ollama Llama 8B (fast, local, sufficient for simple DATA steps)
MODERATE → Groq Llama 70B
HIGH → Groq Llama 70B, temperature=0.1, 2 extra KB examples
UNCERTAIN → skip, emit status=HUMAN_REVIEW

Cross-verification (Prompt C): For MODERATE and HIGH blocks only, a second LLM call (separate context window) checks the SAS→Python pair for 5 known failure modes and returns {equivalent: bool, issues: list[str], confidence: float}. If confidence < 0.75: retry translation with issues appended to the prompt (max 2 retries). If still failing: status=PARTIAL. All outcomes logged to DuckDB llm_audit.
SCC batching: Partitions sharing an scc_id (circular dependency group) are translated together in a single batch — all members provided as context simultaneously, preventing reference errors.
Import detection: Regex scan of generated Python/PySpark code detects which libraries are used (pandas, numpy, statsmodels, scipy, datetime, re, pyspark.sql, etc.) and stores in ConversionResult.imports_detected for ImportConsolidator.
Output: ConversionResult — {conversion_id, partition_id, python_code, target_runtime, imports_detected, status, llm_confidence, failure_mode_flagged, model_used, kb_examples_used, retry_count, validation_result, trace_id}
ValidationAgent (NEW — post-translation quality gate)
Executes the generated Python/PySpark code on a synthetic 100-row pandas (or PySpark) DataFrame to verify runtime correctness before the block proceeds to merge. This addresses a key gap: CodeBLEU and cross-verification measure surface and structural similarity, but neither catches silent runtime errors (e.g., a NameError from a missing variable, a TypeError from wrong column types, or a wrong date offset).
Process:
1. Build a synthetic DataFrame: 100 rows, columns inferred from the SAS DATA step’s variable_scope (numeric → random float, character → random string, date → random date in 2020–2024).
2. Execute the generated Python code via exec() in a sandboxed namespace (restricted builtins, 5-second timeout, no file I/O, no network).
3. Capture the output DataFrame (or error traceback).
4. Apply validation rules:
   • No uncaught exceptions (NameError, TypeError, KeyError, SyntaxError).
   • Output DataFrame has the expected columns (inferred from SAS OUTPUT or DATA statement).
   • For DATE_ARITHMETIC blocks: spot-check that date values are in a plausible range (not offset by 3,653 days — the 1960–1970 epoch gap).
   • For MERGE_SEMANTICS blocks: verify row count is within 0.8–1.5× of input row count (catches Cartesian explode errors).
5. Result: ValidationResult — {validation_id, conversion_id, passed: bool, errors: list[str], warnings: list[str], execution_time_ms: float}.
6. Routing:
   • passed=True → proceed to L4 merge.
   • passed=False AND retry_count < 2 → re-translate with errors appended to prompt.
   • passed=False AND retry_count ≥ 2 → status=PARTIAL, proceed with warning.
   • structural_only blocks (CONDITIONAL_BLOCK, LOOP_BLOCK, etc.) → skip validation, auto-pass.
For PySpark target: uses a local SparkSession (spark.master=local[1], 512 MB driver memory) with a 10-second timeout. Falls back to pandas validation if Spark is unavailable.
4.7 Layer 4 — Merge
ImportConsolidator
Collects imports_detected from all ConversionResult objects for a given file. Deduplicates (each library appears once). Orders per PEP8: stdlib → third-party → local. Maps detected library names to canonical import statements (pandas → import pandas as pd, statsmodels → import statsmodels.api as sm, etc.).
DependencyInjector
Resolves variable name consistency across blocks. Builds a name_registry: dict[str, str] mapping SAS dataset names to Python variable names (snake_case, e.g., WORK.TEMP → work_temp). Scans each PartitionIR.raw_code for datasets produced (DATA {name}) and referenced. Patches the corresponding Python code to use consistent variable names. For unresolvable cross-file references: inserts a comment stub # NOTE: {dataset} expected from external file {source_file}.
ScriptMerger
Assembles the final Python script:

Sort ConversionResult[] by PartitionIR.line_start to preserve original SAS execution order.
For HUMAN_REVIEW or FAILED blocks: insert a structured TODO stub with the original SAS code commented out.
Prepend consolidated import block from ImportConsolidator.
Prepend a header comment: source file, timestamp, block count, partial count.
Run ast.parse(final_script): if SyntaxError is raised, set syntax_valid=False and capture the error location.
Write to output/{source_filename}_converted.py (or _converted_spark.py for PySpark target).
Persist MergedScript to merged_scripts database table.

ReportAgent (NEW — post-merge conversion report)
Generates a structured Markdown and HTML conversion report after ScriptMerger completes each file. The report is the primary deliverable for human reviewers and project managers who need to assess conversion quality without reading code.
Report contents:
• Header: source file path, timestamp, total block count, target runtime (Python / PySpark).
• Summary table: SUCCESS / PARTIAL / FAILED / HUMAN_REVIEW counts with percentages.
• Failure mode breakdown: per-mode counts (DATE_ARITHMETIC: 3, MERGE_SEMANTICS: 1, etc.).
• HUMAN_REVIEW blocks: list of block IDs, partition types, and original SAS snippets (first 10 lines).
• CodeBLEU scores (when gold standard is available): overall and per-partition-type breakdown.
• Validation results: pass/fail/warn counts from ValidationAgent.
• Dependency graph summary: number of nodes, edges, SCC groups detected.
• KB retrieval stats: mean similarity score, hit coverage percentage.
Output: output/{source_filename}_report.md + output/{source_filename}_report.html (via markdown2 conversion).
Persist ConversionReport to conversion_reports DuckDB table.

4.8 Continuous Learning
FeedbackIngestionAgent
Accepts corrections from two sources:

CLI: python scripts/submit_correction.py --conversion_id {uuid} --python_file corrected.py
Automated: when cross-verifier rejects a block after 2 retries, the final retry's SAS+Python pair is automatically submitted

For each correction: runs Prompt C cross-verification on the corrected pair. If confidence ≥ 0.85: creates a new KB example with source="correction", embeds with Nomic Embed, upserts into LanceDB sas_python_examples. Updates the original ConversionResult.status to SUCCESS. Logs to feedback_log DuckDB table.
ConversionQualityMonitor
Runs at the end of every processing batch. Queries DuckDB for the last 100 conversion_results and computes:

success_rate = n_SUCCESS / n_total (target ≥ 0.70)
partial_rate = n_PARTIAL / n_total
avg_llm_confidence (target ≥ 0.75)
failure_mode_dist = Counter of failure_mode_flagged values

Alerts via structlog WARNING if: success_rate < 0.70, avg_llm_confidence < 0.75, or any single failure mode accounts for >40% of PARTIAL conversions (indicates a KB gap for that pattern category).
Retraining Trigger
Monitors four conditions, any of which triggers sklearn LR retraining:

KB grew by ≥500 verified examples since last training
ECE on held-out 20% > 0.12
success_rate < 0.70 for two consecutive batches
A specific failure mode category flagged as KB gap by ConversionQualityMonitor

On KB gap detection: also triggers targeted KB generation (generate_pairs.py for the underrepresented category specifically), rather than full regeneration.

5. Knowledge Base
5.1 Design Rationale
The knowledge base (KB) is the primary quality driver for translation accuracy. Every LLM call is augmented with retrieved SAS→Python/PySpark pairs before generation, meaning the quality of the KB directly determines the quality of the output. A KB with wrong examples is worse than no KB at all.
Key decision: dual-LLM synthetic generation with cross-verification. The KB is built without any human SAS expertise using a 3-prompt chain: Prompt A generates a realistic SAS block, Prompt B converts it to Python (and optionally PySpark) with explicit failure-mode rules in the prompt, Prompt C cross-verifies equivalence and returns structured JSON. Pairs with confidence ≥ 0.85 are accepted with verified=True. This removes the dependency on SAS runtime access or SAS expertise while maintaining quality control.
KB Versioning: Every KB example carries a version integer (starting at 1). When a correction replaces an existing pair, the old pair is retained with superseded_by pointing to the new example_id, and the new pair gets version = old_version + 1. A dedicated kb_changelog DuckDB table logs every mutation (insert, update, supersede) with timestamp, author ("llm_gen" or "human_correction"), and diff summary. This enables: (1) rollback to any previous version of a specific pair, (2) audit trail for KB quality drift analysis, (3) A/B retrieval experiments comparing old vs. new pairs. Rollback: python scripts/kb_rollback.py --example_id {uuid} --to_version {n} restores the specified version and marks the current one as superseded.

7.4 NetworkX Graph Structure
The dependency graph is built in-memory using NetworkX (DiGraph). Each partition is a node with attributes:
  partition_id, partition_type, risk_level, complexity_score, file_id, scc_id

Two edge types are stored as edge attributes (edge_type key):
  DEPENDS_ON — dep_type: "dataset" | "libref" | "join_input"
  MACRO_CALLS — macro_name: the called macro identifier

Traversal uses nx.ancestors() / nx.descendants() bounded by max_hop.
SCC detection uses nx.strongly_connected_components() + nx.condensation().

8. Technology Stack — With Benchmarking Justification
Each technology selection is justified against at least one concrete alternative, with measurable criteria.

8.1 LLMs — Local Inference
Llama 3.1 8B via Ollama — boundary resolution, LOW-risk translation
Chosen over:

GPT-4o: costs $0.005/1K tokens; a 500K-line SAS project generates ~50M tokens in translation = $250/run. Not viable for a student project.
Mistral 7B: similar capability tier, but Llama 3.1 has stronger performance on structured code tasks per HuggingFace Open LLM Leaderboard (Aug 2024).
LlamaCpp: Ollama provides the same GGUF quantized inference with a simpler REST API and automatic GPU/CPU routing.

Benchmarks:

Latency: ~200ms/call on CPU, ~80ms on GPU
Context window: 128K tokens (sufficient for any SAS block)
Memory: 4–5 GB RAM for 4-bit quantized Q4_K_M
Code instruction-following: adequate for simple DATA steps and PROC blocks with few-shot examples


Llama 3.1 70B via Groq API — cluster summarization, MODERATE/HIGH translation
Chosen over:

Local 70B: requires 40+ GB VRAM (A100 or 2×3090) — not available on a student machine
Claude API / GPT-4: paid APIs; Groq free tier provides Llama 70B with rate limits sufficient for this project
Llama 3.1 8B for HIGH blocks: empirically, 8B models hallucinate on nested macro chains and PROC SQL with 3-level subqueries; 70B models are measurably better on complex multi-step reasoning tasks (MMLU Pro, HumanEval+)

Benchmarks (Groq free tier):

Rate limit: 30 requests/minute, 6,000 tokens/minute
Latency: ~800ms/call
HumanEval: Llama 3.1 70B scores 80.5% vs 8B's 72.6%
Three-tier fallback: Groq → Ollama 70B local → heuristic (no LLM required)


8.2 Embedding Model
Nomic Embed v1.5 (nomic-ai/nomic-embed-text-v1.5)
Chosen over:

OpenAI text-embedding-ada-002: costs $0.0001/1K tokens; embedding 50K blocks = 500K tokens = $50/run. Local is always better when viable.
all-MiniLM-L6-v2: 384-dim only. Too low-dimensional for the diversity of SAS code patterns; MTEB score 56.26 vs Nomic's 62.39.
bge-large-en-v1.5: 768-dim, strong MTEB (64.23), but 335M parameters (3× larger than Nomic) with no meaningful accuracy gain on code retrieval tasks.

Benchmarks:

MTEB overall score: 62.39 (vs ada-002: 60.99)
Dimensions: 768 — sufficient for GMM clustering without dimensionality reduction
License: Apache 2.0 — commercial use permitted
Inference: 50ms/block on CPU, 15ms on GPU
Model size: 137M parameters, ~550 MB RAM


8.3 Vector Store
LanceDB
Chosen over:

FAISS: no persistence (in-memory only unless you build a custom persistence layer); no metadata filtering; no cloud sync.
Pinecone / Weaviate: cloud-hosted (requires internet, account, billing); not deployable locally.
ChromaDB: similar capability tier, but LanceDB's columnar on-disk format (based on Apache Lance) provides 10–100× faster filtered vector search vs ChromaDB's SQLite-backed index on datasets > 100K vectors (LanceDB benchmark, 2024).
pgvector (PostgreSQL extension): viable but requires PostgreSQL running as a service; LanceDB is embedded (like DuckDB — zero-ops).

Benchmarks (LanceDB internal, 2024):

Query latency @1M vectors: 2–5ms (IVF cosine index)
Filtered search: 10× faster than FAISS on metadata-filtered queries
License: Apache 2.0
Storage: columnar Lance format on local disk — no server required


8.4 Graph Library
NetworkX (in-memory)
Chosen over:

Neo4j: GPL license in Community Edition; requires a JVM server process; 2+ hours to set up; significant ops overhead for a student project.
DGL (Deep Graph Library): designed for graph neural networks, not property graph queries.
ArangoDB: multi-model (document + graph), requires server process.
KuzuDB: project abandoned (Oct 2025, archived on GitHub). Was initially selected but removed due to maintenance risk.

Why NetworkX is sufficient:

In-memory DAG operations: topological sort, SCC condensation, ancestor/descendant traversal — all available natively.
Partition graphs are small (< 50K nodes for enterprise codebases) — fits entirely in memory.
License: BSD 3-Clause.
No server, no setup, no external dependency beyond pip install.
nx.ancestors() bounded by max_hop replaces Cypher multi-hop traversal.
Latency: < 1ms for ancestor queries on 10K-node graphs.


8.5 Graph Algorithm Library
NetworkX (Python)
Used for: topological sort, SCC detection (strongly_connected_components), longest path computation (dag_longest_path_length), in-memory DAG representation.
Chosen over:

igraph: faster for large graphs (C backend), but the Python API is less intuitive and the API surface for SCC + condensation is more complex.
Networkit: designed for billion-edge graphs; massive overkill for SAS dependency graphs (typical: 100–10,000 nodes).
CuGraph (RAPIDS): GPU-accelerated, requires CUDA — not portable.

For a SAS dependency graph of 100–10,000 partitions, NetworkX is fast enough (SCC on 10,000 nodes: < 100ms) and has the most complete algorithm library.

8.6 Parsing and Grammar
Lark (LALR parser)
Chosen over:

Hand-coded regex: regex cannot handle nested structures (nested %IF/%DO inside %MACRO inside PROC SQL) without exponential backtracking. Regex is used for the fast pre-scan in CrossFileDependencyResolver (no nesting required there); Lark handles the structural parsing.
ANTLR4: more powerful (full LL(*) parsing), but significantly more complex to set up (Java runtime, grammar compilation step). Lark compiles grammars on-the-fly in pure Python with no external tools.
ply (PLY): older library, LALR-only, less maintained, less clear error messages.

Benchmarks:

Parse speed: ~10,000 lines/second on a standard SAS grammar (measured on test corpus)
License: MIT
Grammar format: EBNF-like, human-readable, version-controlled


8.7 Complexity Analysis
Radon
Chosen over:

Manual AST walking: would require implementing McCabe complexity from scratch for SAS code — weeks of work with no benchmark guarantee.
Lizard: multi-language complexity tool, but handles SAS as a generic language (no SAS-specific constructs recognized). Radon's Python-focused output is accurate for the parts of SAS that can be analyzed as pseudo-code; we supplement with custom features (macro density, nesting depth) for the SAS-specific elements.

Radon is used only for the cyclomatic_complexity feature. The other 4 features (nesting depth, macro density, cross-file deps, SQL complexity) are computed with custom logic in ComplexityAgent.

8.8 ML Framework
scikit-learn (Logistic Regression + GaussianMixture + CalibratedClassifierCV)
Chosen over:

PyTorch / Keras neural network: overkill for a 5-feature classification problem with 500 training examples. A neural network would overfit and produce worse-calibrated probabilities than a properly calibrated LR. The ECE of a neural network softmax output is well-documented to be poorly calibrated without explicit calibration steps.
XGBoost: better than LR on complex nonlinear data, but with 500 examples and 5 features, the complexity boundary is not complex enough to justify. Interpretability of LR coefficients is a defense advantage.
Hardcoded thresholds: no probabilistic confidence, no calibration, no retraining capability.

Platt scaling (CalibratedClassifierCV(method='sigmoid', cv=5)) converts raw LR log-odds to calibrated probabilities. ECE measures how well the confidence scores correspond to actual accuracy — a model with ECE=0.08 is 8% off in probability on average, which is the threshold for trusting the calibration_confidence < 0.65 → HUMAN_REVIEW rule.

8.9 Orchestration
LangGraph
Chosen over:

Custom FSM: reimplementing state management, error recovery, and fan-out from scratch is weeks of work with no research value. LangGraph provides these primitives out of the box.
LangChain (without LangGraph): LangChain's sequential chains are not suitable for the branching, parallel, and checkpoint-resume patterns required. LangGraph's StateGraph is purpose-built for complex agentic workflows.
Prefect / Airflow: workflow orchestrators designed for batch data pipelines with scheduled execution — not for in-process agentic loops with dynamic branching and LLM calls.
Celery: task queue for distributed workers — appropriate for scaling to a production cluster but excessive complexity for a 4-month student project.


8.10 Checkpointing
Redis
Chosen over:

SQLite only: every checkpoint write would lock the SQLite database (single-writer), creating a bottleneck on concurrent processing of multiple files. Redis supports atomic SETEX with no lock contention.
File system: writing checkpoint JSON to disk is viable but slow (disk I/O) and requires custom cleanup logic. Redis TTL (24h) handles cleanup automatically.
In-memory dict: lost on crash — defeats the purpose.

Redis is used exclusively for fault-tolerant checkpointing. The SQLite store remains the authoritative partition IR.

8.11 Analytics and Audit
DuckDB
Chosen over:

Pandas (in-memory): cannot persist across runs; no SQL query capability; poor performance on aggregations over 100K+ audit records.
PostgreSQL OLAP: requires a server process; same ops overhead as Neo4j — excessive for an analytics sidecar.
SQLite: OLAP queries (window functions, groupby aggregations over audit logs) are significantly slower in SQLite than DuckDB. DuckDB is specifically optimized for analytical queries, not transactional writes.

DuckDB is used for read-heavy analytics tables (llm_audit, calibration_log, ablation_results, quality_metrics, feedback_log). All writes are append-only, playing to DuckDB's columnar strengths.

8.12 Async I/O
aiofiles
The only viable option for async file reading in Python without a thread pool. asyncio itself does not support async file I/O natively (files are not sockets). aiofiles wraps file operations in a thread executor transparently, providing an async/await interface. Used in StreamAgent for the 8 KB async generator.

8.13 Typed LLM Output
Instructor
Chosen over:

Manual JSON parsing: LLMs frequently produce malformed JSON, markdown-wrapped JSON, or partial JSON. Retry logic for JSON parsing errors adds significant complexity.
LangChain's PydanticOutputParser: tightly coupled to LangChain's prompt template system; less flexible for our Ollama + Groq dual-endpoint setup.
Outlines (structured generation): requires access to model logits (server-side), not available with Groq's API.

Instructor wraps any OpenAI-compatible API (both Ollama and Groq are compatible) and enforces a Pydantic model as the output type, retrying automatically on validation failure. Used in LLMBoundaryResolver and the cross-verifier.

8.14 Data Validation
Pydantic v2
All data models (PartitionIR, RAPTORNode, ConversionResult, etc.) use Pydantic v2. Chosen because: Pydantic v2 (Rust-backed) is 5–50× faster than v1 at validation; it is the de-facto standard for FastAPI and LangGraph; model.model_dump() produces clean JSON for serialization; field validators provide centralized data integrity enforcement.

8.15 Logging
structlog
Chosen over:

Python stdlib logging: produces line-based text output, making it difficult to query log fields programmatically. No native JSON output without custom formatters.
Loguru: simpler API, but structlog's structured context binding (every log line carries trace_id, agent_name, partition_id) is critical for debugging LLM calls that produce wrong output.

structlog outputs JSON lines to both stdout and a rotating file (logs/partition.log). Every log record carries: timestamp, level, agent, trace_id, event, and a data dict with structured fields. This is directly queryable with DuckDB's read_json_auto().

9. Agent Architecture
9.1 Design Principle
Agents are not functions. Each of the 14 agents has:

Autonomous decision-making logic (not just a wrapper around a library call)
Retry / fallback behaviour with exponential backoff
Typed Pydantic input and output contracts
Structured logs with trace_id + partition_id on every record
Inherits from BaseAgent abstract class

Helper utilities (the backpressure queue wrapper, the tiktoken guard, the embedding hash function) are plain functions — not agents. This discipline keeps the count at 14 real agents versus the inflated 20+ seen in naive designs.

10.1 Boundary Accuracy Benchmark
Dataset: 50 SAS files manually annotated with gold-standard block boundaries. Each file contains 2–5 annotated blocks → ~150 block-level examples total (the knowledge_base/gold_standard/ corpus).
Metric: Boundary accuracy = correctly detected blocks / 150. A boundary is "correct" if partition_type matches AND line_start is within ±2 lines AND line_end is within ±2 lines of the gold annotation.
Target: > 90% (spec v4.1 upgrade from the naive design's 85%)
CI guard: tests/regression/test_boundary_accuracy.py asserts > 0.90 on every push.
10.2 RAPTOR Retrieval — Ablation Study
Setup: The 50 benchmark files, each queried with 10 synthetic retrieval questions (→ 500 queries total). Each query is run against both the flat-index (leaf embeddings only) and the RAPTOR index (leaf + cluster + root). Results stored in DuckDB ablation_results.
Metrics:

Hit-rate@5: is the ground-truth relevant block in the top-5 retrieved results?
MRR (Mean Reciprocal Rank): 1/rank if found, 0 otherwise
p50 query latency (ms)
Hit-rate@5 stratified by complexity tier (LOW / MODERATE / HIGH)

Target: RAPTOR hit-rate@5 > 0.82, RAPTOR > flat by ≥10% on MODERATE/HIGH blocks.
Research claim: this ablation directly validates the RAPTOR adaptation. If RAPTOR outperforms flat on complex blocks but not on simple ones, this matches the theoretical prediction — RAPTOR adds value precisely when context grouping matters.
10.3 Calibration — ECE
Metric: Expected Calibration Error (ECE). Measures how well the calibration_confidence values produced by StrategyAgent correspond to actual prediction accuracy. ECE=0 is perfect calibration; ECE=0.10 means the model's confidence is off by 10% on average.
Measurement: Held-out 20% of the 500-example training set. Computed via reliability diagram and bin-averaged calibration error.
Target: ECE < 0.08 (tightened from 0.10 in earlier versions; enforced via CI guard tests/regression/test_ece.py).
Why it matters for the jury: the calibration_confidence < 0.65 → HUMAN_REVIEW rule only works if the confidence values are meaningful. If ECE > 0.15, a block with confidence=0.70 might actually be correct only 55% of the time — the system would route it to automated translation when it should go to human review. ECE < 0.08 makes the threshold defensible.
10.4 Differential Testing
For test_coverage_type="full" blocks, generated Python is tested by: (1) creating a synthetic 100-row pandas DataFrame, (2) running the Python against it, (3) verifying the output matches expectations for the SAS construct being tested.
Coverage target: 45% of all blocks (only full-coverage type blocks are testable; structural_only blocks are syntax-checked only via ast.parse()).
Extended test cases:

Nested macro parameter passing (up to depth 3)
%DO/%WHILE with counter variable injection
PROC SQL with correlated subqueries (2-level nested SELECT)
Multi-table joins (INNER, LEFT, FULL OUTER) on 100-row frames
Edge cases: empty dataset, single-row input, all-null columns, duplicate keys

10.5 Translation Quality
Tracked by ConversionQualityMonitor via DuckDB quality_metrics:

success_rate (target ≥ 0.70 at Week 10, improving with KB growth)
avg_llm_confidence (target ≥ 0.75)
failure_mode_dist (no single mode > 40% of PARTIALs)

End-to-end validation on the two example scripts from the project document (ETL customer script, sales analysis script): both must produce syntactically valid Python that passes ast.parse().

12. Methodology
12.1 Development Methodology
Hybrid Kanban + Weekly Sprints. Chosen for a solo developer on a bounded 14-week project:

Why not pure Scrum: No standup overhead for a solo developer. The sprint ceremony is a team coordination mechanism — irrelevant here.

Why CodeBLEU, Not BLEU
Before the per-layer breakdown, this distinction matters for the jury.
BLEU (Papineni et al., 2002) was designed for natural language translation. It measures n-gram overlap between a candidate and reference translation. Applied to code, it is a poor metric for three reasons: (1) variable names don't need to match exactly (a correct translation might use df_customers where the reference uses customers — BLEU penalizes this), (2) it treats all tokens equally (the keyword def has the same weight as an arbitrary string), and (3) it ignores syntactic structure (two strings with the same tokens in different order have the same BLEU but one might be a SyntaxError).
CodeBLEU (Ren et al., 2020) fixes this for code by combining four weighted sub-scores:
Sub-scoreWeightWhat it Measuresn-gram BLEU0.25Token-level overlap (surface similarity)Weighted n-gram0.25Keyword-aware overlap (Python keywords weighted higher than identifiers)AST Match0.25Syntactic structure similarity (tree edit distance on ASTs)Data-flow Match0.25Semantic equivalence via data-flow graph comparison
CodeBLEU is used specifically for Layer 3 (Translation) evaluation where a reference Python conversion exists (gold standard corpus). For layers where code quality is not the output (L2-A through L2-E), domain-specific metrics apply.
Reference: Ren, S., Guo, D., Lu, S., Zhou, J., Liu, S., Tang, D., ... & Ma, S. (2020). CodeBLEU: a Method for Automatic Evaluation of Code Synthesis. arXiv:2009.10297.

How to Present CodeBLEU to the Jury
If a jury member challenges CodeBLEU as "not a real execution test," this is the prepared response:

"CodeBLEU is not a substitute for execution testing — it is a complementary metric. We use it because: (1) we have no SAS runtime to execute the source programs against, which means true input/output comparison is not possible for the SAS side; (2) CodeBLEU's AST sub-score catches structural errors that BLEU misses; (3) our differential testing covers the subset of blocks where we can simulate execution on synthetic DataFrames. The two metrics address different failure modes: CodeBLEU catches structural and surface-level translation errors; differential testing catches semantic equivalence failures on the specific SAS constructs where we expect LLMs to fail. Neither alone is sufficient; together they form a defensible evaluation strategy consistent with the code generation literature (Ren et al. 2020, Lu et al. 2021, Chen et al. 2021)."

Additional references to cite alongside CodeBLEU:

Lu, S. et al. (2021). CodeXGLUE: A Machine Learning Benchmark Dataset for Code Understanding and Generation. NeurIPS 2021. — defines the benchmark suite where CodeBLEU is standard.
Chen, M. et al. (2021). Evaluating Large Language Models Trained on Code. arXiv:2107.03374. — HumanEval, the execution-based complement to CodeBLEU.
Austin, J. et al. (2021). Program Synthesis with Large Language Models. arXiv:2108.07732. — MBPP, another execution-based code eval benchmark.
