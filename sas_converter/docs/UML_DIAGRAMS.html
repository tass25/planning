How to read this diagram

Everything starts with BaseAgent at the top — it's an abstract class, meaning you can never create a BaseAgent directly. Instead, it defines the contract that every agent must follow: each agent gets a unique trace_id (for logging and debugging), a structured logger, and must implement two methods: agent_name() (returns a human-readable name like "FileAnalysisAgent") and process() (the main work method, called asynchronously).

  1. FileAnalysisAgent inherits from BaseAgent. When you call its process(project_root), it walks the project folder, analyzes every .sas file it finds, and returns a list of FileMetadata objects (the dotted arrow labeled "produces"). FileMetadata is a Pydantic model — a validated data container holding the file path, detected encoding (UTF-8, Latin-1, etc.), the SHA-256 hash of the file content, line count, and whether the file passed Lark pre-validation.
  2. RegistryWriterAgent takes those FileMetadata objects and writes them to the database as FileRegistryRow records (SQLAlchemy ORM model). It uses INSERT OR IGNORE so that if the same file (same content_hash) is already in the database, it silently skips it — that's how deduplication works.
  3. CrossFileDependencyResolver reads the SAS source code of each file and uses regex to find %INCLUDE and LIBNAME references — these are how SAS files reference other files. For each reference found, it tries to resolve the path and writes a CrossFileDependencyRow to the database. The "1" to "*" arrow from FileRegistryRow means one file can have many cross-file dependencies.
  4. DataLineageExtractor also reads the source code but looks for data flow statements: which tables a SAS program reads from (SET, MERGE, FROM, JOIN) and which tables it writes to (DATA, CREATE TABLE, INSERT INTO). Each match becomes a DataLineageRow in the database.

The three enum classes on the right (PartitionType, RiskLevel, ConversionStatus) are used by PartitionIR — the intermediate representation of a single code block. Although PartitionIR isn't created by the L2-A agents directly, it's shown here because it's the data model that downstream agents (Week 3+) will use to represent each detected SAS block.

Key takeaway: the 4 agents share the same interface (BaseAgent), but each one has a single responsibility — scan files, register them, resolve dependencies, or extract lineage. They communicate through Pydantic models (in-memory) and SQLAlchemy rows (persisted to SQLite).

Walking through the pipeline step by step

This diagram reads top to bottom as a timeline. Each vertical line is a participant (an agent or the database), and horizontal arrows are messages between them. Here's what happens when you run the L2-A pipeline:

  1. Step 1 — File Discovery. The user (or orchestrator) calls FileAnalysisAgent.process(project_root). The agent first runs rglob("*.sas") — that's Python's recursive glob, which walks every subdirectory under project_root and finds all files ending in .sas. Then, for each file found (that's the loop box), it does three things: (a) detects the character encoding using the chardet library (because SAS files can be UTF-8, Latin-1, Windows-1252, etc.), (b) computes the SHA-256 hash of the raw bytes (this unique fingerprint is used later for deduplication), and (c) runs a quick regex pre-validation to catch obvious syntax issues. The result is a list of FileMetadata objects returned to the caller.
  2. Step 2 — Registration. The user passes that list to RegistryWriterAgent.process(files, engine). For each file, it asks the database: "Do you already have a file with this content_hash?" If yes, the file is a duplicate — skip it. If no, insert a new FileRegistryRow with status PENDING. This prevents re-processing the same file twice, even if it appears in different folders. At the end, it commits the transaction and reports how many files were inserted vs. skipped.
  3. Step 3 — Dependency Resolution. Next, CrossFileDependencyResolver.process() runs. For each file, it scans the source code with 3 regex patterns looking for %INCLUDE, LIBNAME, and variable-based %INCLUDE &var references. For each match, it calls _resolve(ref_path) to see if the referenced file actually exists in the project. Whether resolved or not, it inserts a CrossFileDependencyRow into the database. The return value tells you how many references were found, how many resolved successfully, and how many remain unresolved.
  4. Step 4 — Lineage Extraction. Finally, DataLineageExtractor.process() scans each file's source code with two groups of regex patterns: read patterns (SET, MERGE, FROM, JOIN) to find which datasets the code reads from, and write patterns (DATA, CREATE TABLE, INSERT INTO) to find which datasets it writes to. Each match is inserted as a DataLineageRow. The return value summarizes the total reads and writes found.

Notice that the agents run sequentially — each one finishes before the next starts. This is intentional: you first need to discover files (Step 1) before you can register them (Step 2), and you need registered files before you can resolve their dependencies (Step 3) or extract lineage (Step 4).

The alt box in Step 2 shows a conditional branch: the agent takes a different path depending on whether the file is new or a duplicate. This is the deduplication logic that prevents the same SAS file from appearing twice in the registry.

The big picture — from raw SAS to converted Python

This diagram shows the complete 16-agent architecture across 6 processing layers. Think of it as an assembly line: raw .sas files enter on the left and fully converted .py files come out on the right, with each layer adding more understanding.

  1. L2-A: Entry & Scan (green = implemented) — The 4 agents we built in Week 1–2. They discover files, register them in the database, map cross-file dependencies, and extract data lineage. This is the foundation everything else builds on.
  2. L2-B: Streaming Core (yellow = planned) — StreamAgent will read large SAS files in small 8 KB chunks using async I/O, so we never load a 100 MB file into memory at once. StateAgent maintains a finite-state machine that tracks what kind of code block we're inside (DATA step? PROC? macro?), how deep the nesting is, and which macros are on the stack.
  3. L2-C: RAPTOR Chunking (yellow = planned) — This is the core innovation. BoundaryDetectorAgent uses a Lark LALR grammar to detect where one SAS block ends and another begins (handles ~80% of cases deterministically). The remaining ~20% ambiguous boundaries are sent to a local LLM (Llama 3.1 8B) for resolution. PartitionBuilderAgent builds a PartitionIR object for each detected block (one of 9 canonical types). Then RAPTORPartitionAgent embeds all blocks using Nomic Embed, clusters them with GMM, summarizes each cluster with Llama 70B, and builds a hierarchical RAPTOR tree — this is what makes retrieval work better than flat partitioning.
  4. L2-D: Complexity & Strategy (yellow = planned) — ComplexityAgent computes 5 features for each block (including cyclomatic complexity via radon) and a calibrated logistic regression model assigns a risk level (LOW / MODERATE / HIGH). StrategyAgent decides the translation strategy based on risk: simple blocks get rule-based translation, complex blocks get LLM-assisted translation.
  5. L3: Translation (yellow = planned) — TranslationAgent queries the knowledge base for similar SAS→Python examples, then uses an LLM to generate the Python/PySpark equivalent. ValidationAgent checks the generated code via ast.parse() and runs it against a synthetic DataFrame to verify correctness.
  6. L4 + CL: Merge & Learning (yellow = planned) — ReportAgent assembles the final converted script, merges imports, and generates a conversion report. FeedbackIngestionAgent and ConversionQualityMonitor form the continuous learning loop: human corrections feed back into the KB, and quality metrics trigger model retraining if accuracy drops.

At the bottom, 3 storage systems support the pipeline: SQLite for the file registry and lineage data, Kuzu (an embedded graph database) for the dependency DAG, and DuckDB for analytics and audit logging.

The green layer (L2-A) is what we have today — 4 working agents with 20 passing tests. The yellow layers are the roadmap for Weeks 3–14. Each layer builds on the previous one, so the pipeline grows incrementally.

4. Entity-Relationship Diagram — Database Schema

Shows the 3 SQLite tables and their foreign key relationships. WAL mode and FK enforcement are enabled at engine level.

erDiagram FILE_REGISTRY { string file_id PK string file_path string encoding string content_hash UK int file_size_bytes int line_count boolean lark_valid text lark_errors string status text error_log string created_at } CROSS_FILE_DEPS { int id PK string source_file_id FK string ref_type string raw_reference boolean resolved string target_file_id FK string target_path } DATA_LINEAGE { int id PK string source_file_id FK string lineage_type string source_dataset string target_dataset text source_columns string target_column string transform_expr int block_line_start int block_line_end } FILE_REGISTRY ||--o{ CROSS_FILE_DEPS : "source_file_id" FILE_REGISTRY ||--o{ CROSS_FILE_DEPS : "target_file_id" FILE_REGISTRY ||--o{ DATA_LINEAGE : "source_file_id"

How the data is stored

The system uses a single SQLite database (with WAL mode enabled for concurrent reads) containing 3 tables. Here's how they relate:

  1. FILE_REGISTRY is the central table — every SAS file discovered by the pipeline gets exactly one row here. The file_id (a UUID) is the primary key, and content_hash has a UNIQUE constraint, which is how deduplication works: if two files have identical content, only the first one is stored. Other columns track the file path, detected encoding, line count, whether it passed Lark pre-validation (lark_valid), and a status field that tracks where the file is in the processing pipeline (PENDING, PROCESSING, etc.).
  2. CROSS_FILE_DEPS stores cross-file references found by the CrossFileDependencyResolver. Each row links a source file to a target file via source_file_id and target_file_id (both foreign keys pointing to FILE_REGISTRY). The ref_type column says what kind of reference it is (%INCLUDE, LIBNAME, etc.), raw_reference stores the original text from the SAS code, and resolved tells you whether the system was able to find the target file in the project. Notice the two relationship lines from FILE_REGISTRY — one file can be the source of many dependencies, and also the target of many dependencies.
  3. DATA_LINEAGE stores data flow information found by the DataLineageExtractor. Each row represents one "table A is read/written by file X" relationship. The lineage_type says whether it's a read operation (TABLE_READ: SET, MERGE, FROM, JOIN) or a write operation (TABLE_WRITE: DATA, CREATE TABLE, INSERT INTO). The source_dataset and target_dataset columns identify the SAS datasets involved, and block_line_start / block_line_end pinpoint exactly where in the source code this data flow occurs.

The crow's-foot notation on the relationship lines reads as: one FILE_REGISTRY record can have zero or many (||--o{) CROSS_FILE_DEPS or DATA_LINEAGE records. This is a classic one-to-many relationship.

Foreign key enforcement is turned on explicitly at engine creation time (PRAGMA foreign_keys = ON). This means SQLite will reject any INSERT into CROSS_FILE_DEPS or DATA_LINEAGE if the referenced file_id doesn't exist in FILE_REGISTRY — ensuring data integrity.

The life of a SAS file

This diagram tracks every possible state a SAS file goes through from the moment it's discovered to the moment it's fully converted. Think of each box as a status badge attached to the file.

  1. DISCOVERED — The file has been found by FileAnalysisAgent on disk. At this point, we know its path, encoding, and hash, but nothing else. From here, two things can happen:
    • If the file fails pre-validation (unclosed macros, truncated PROCs, etc.), it moves to INVALID, and then to HUMAN_REVIEW because a human needs to fix the syntax error in the original SAS file.
    • If it passes, RegistryWriterAgent inserts it into the database and the state becomes REGISTERED. Notice the nested sub-state inside REGISTERED: the file first becomes PENDING (waiting to be processed), then PROCESSING (actively being worked on).
  2. DEPS_RESOLVED — CrossFileDependencyResolver has finished scanning this file for %INCLUDE/LIBNAME references. We now know which other files it depends on.
  3. LINEAGE_EXTRACTED — DataLineageExtractor has mapped all the data flow (which tables are read, which are written). At this point, L2-A is complete for this file.
  4. STREAMED → PARTITIONED → SCORED — These are the L2-B through L2-D stages (Weeks 2–4). The file is streamed in chunks, split into individual code blocks, and each block is scored for complexity.
  5. TRANSLATING → VALIDATED → MERGED → COMPLETE — The L3 and L4 stages (Weeks 10–11). Each block is translated to Python, validated, merged into a final script, and the file reaches its terminal state: COMPLETE.

There's also an error path: if the LLM fails during translation, the file moves to FAILED, then HUMAN_REVIEW. Once a human fixes the issue, the file can be re-submitted and goes back to REGISTERED to try again.

Currently (Week 1–2), only the states from DISCOVERED through LINEAGE_EXTRACTED are active in the codebase. The later states (STREAMED onward) will be implemented in Weeks 3–11 as each new layer is built.

How we validate the system's accuracy

The gold standard corpus is our ground truth — 50 hand-annotated SAS files where a human expert has marked exactly which blocks exist and what data lineage each file contains. This diagram shows the automated validation loop that checks whether our agents produce the same results.

  1. Load a .sas file from the knowledge_base/gold_standard/ folder. For example, gs_simple_read.sas.
  2. Load the matching .gold.json file (gs_simple_read.gold.json). This JSON file contains the human-annotated truth: the expected list of blocks (with their types and line ranges) and the expected data lineage entries (which tables are read and written).
  3. Run FileAnalysisAgent on the SAS file to get a FileMetadata object (path, encoding, hash, etc.). This is the same agent used in production — we're testing the real code.
  4. Run DataLineageExtractor on the file content to find all TABLE_READ and TABLE_WRITE operations. The extractor returns a list of DataLineageRow objects.
  5. Compare blocks: We check the detected block types and line ranges against the expected_blocks array in the .gold.json. If they match → green check (correct). If not → red X (the agent made an error, and we log the discrepancy for investigation).
  6. Compare lineage: We check the detected data lineage (source/target datasets, operation types) against the data_lineage section in the .gold.json. Same pass/fail logic.
  7. Loop: This process repeats for all 50 files in the gold standard. After the last file, a validation report is generated summarizing the overall accuracy.

The 50 gold standard files are split into 3 difficulty tiers: Simple (~15 files, 7–50 lines, single block types), Medium (~20 files, 100–250 lines, mixed blocks with macros), and Hard (~15 files, 400+ lines, enterprise ETL with nested macros and CALL EXECUTE). Together they contain 721 annotated blocks, giving us a robust benchmark for measuring accuracy.

This validation flow runs as part of our test suite (pytest tests/). The 20 currently passing tests exercise the FileAnalysisAgent, CrossFileDependencyResolver, RegistryWriterAgent, and DataLineageExtractor against representative gold standard files. As more agents are built, more validation checks will be added.

How the developer interacts with optimization suggestions

This diagram captures the complete lifecycle of a conversion job from the developer's perspective. The key innovation is the optimization suggestion review loop, which puts the developer in control of code quality decisions.

  1. Upload: The developer drags SAS files into the React frontend. The frontend sends them to the FastAPI gateway, which kicks off the full 16-agent pipeline.
  2. Translation + Analysis: After TranslationAgent converts SAS to Python, the OptimizationSuggester runs static analysis to find improvement opportunities — things like loop vectorization, pandas idioms, or missing type hints.
  3. Review in Monaco Editor: The frontend presents a side-by-side code view (SAS left, Python right) with suggestion cards anchored to specific line ranges. Each card shows the file, line range, reason, and confidence score.
  4. Accept/Reject: The developer clicks Accept to apply the suggestion (the code is patched in-place) or Reject to keep the original. Both decisions are logged.
  5. Feedback Loop: Every decision feeds into the FeedbackIngestionAgent. Accepted suggestions become positive training signals; rejected ones become negative signals. Over time, the system learns which suggestions developers actually want.
  6. Report: The final report includes all optimization decisions with their outcomes, giving a complete audit trail.
The WebSocket connection enables real-time progress updates during conversion. The developer sees each stage (scanning → partitioning → translating → validating → optimizing) update live.

8. State Diagram — Optimization Suggestion Lifecycle

How a suggestion moves through the system

Each optimization suggestion is a first-class entity with a tracked lifecycle. This state machine ensures no suggestion is lost and every developer decision is captured for continuous improvement.

  1. Generated: The OptimizationSuggester produces a suggestion after analyzing translated Python code. It includes the exact file path, line range (e.g., AA.py L42–L58), the original code, the suggested replacement, a human-readable reason, and a confidence score (0–1).
  2. Pending: The suggestion is queued in the frontend, waiting for the developer to review it. Multiple suggestions can be pending simultaneously.
  3. Reviewing: The developer opens the suggestion card in the Monaco editor. They see the original code highlighted and the suggested replacement in a diff view.
  4. Accepted → Applied: If the developer clicks Accept, the CodePatcher applies the diff to the output file. The patched code is validated (re-run through ValidationAgent if needed).
  5. Rejected: If the developer clicks Reject, the original code is kept. The rejection reason is optionally captured.
  6. Deferred: If the developer clicks Later, the suggestion returns to Pending. This is useful for suggestions that need more investigation.
  7. Feedback: Both accepted and rejected decisions are logged and sent to the FeedbackIngestionAgent, which maintains the continuous learning signal.
The admin dashboard shows aggregate statistics: acceptance rate per suggestion type, most-rejected patterns, and confidence calibration (are high-confidence suggestions actually accepted more often?).
