Global Architecture

Three-tier stack deployed on Azure Container Apps with CI/CD via GitHub Actions.

Codara — Cloud Architecture

This platform transforms legacy statistical programs into modern, equivalent code while preserving the original logic with mathematical certainty. A user uploads their files through a web interface, which passes them to a secure processing service that orchestrates a sequence of over twenty specialised intelligent agents — each responsible for a different aspect of the conversion, from parsing the structure of the original code, to finding relevant translation examples in a knowledge base, to generating the output and verifying its correctness. Three separate artificial intelligence providers are chained together in a resilience pattern: if one becomes unavailable or rate-limited, the system transparently switches to the next without interrupting the process. Once a translation is produced, it undergoes formal mathematical verification using a theorem prover and adversarial stress-testing with synthetic edge-case data, ensuring it behaves identically to the original under all possible inputs. Four purpose-built databases support the journey — one for user accounts and job tracking, one for pipeline crash recovery, one for intelligent example retrieval using vector similarity, and one for operational analytics and auditing. The entire system is continuously tested, benchmarked against a gold-standard corpus, scanned for vulnerabilities, and deployed through a fully automated pipeline that enforces quality at every stage before reaching production.

Why These Technologies?

Why LangGraph (not LangChain)?
Explicit state machines with typed state. Each node reads/writes shared pipeline state, enabling deterministic execution, per-node checkpointing, and mid-pipeline recovery.
Pipeline recovers from LLM failures without restarting.
Why FastAPI (not Flask)?
All agents are asynchronous. FastAPI's native async support and BackgroundTasks allow running the full pipeline in-process without an external task queue like Celery.
No external task queue needed. SSE streaming built-in.
Why 3 databases (SQLite + LanceDB + DuckDB)?
SQLite: ACID for users/conversions. LanceDB: vector similarity search (768-dim). DuckDB: columnar analytics on LLM logs (100x faster for aggregations).
Best performance per access pattern, zero operational overhead.
Why 3 LLM Providers?
Resilience: fallback chain with circuit breakers. Cross-verification: translate on one provider, verify on another. Cost: local Ollama is free; Groq offers 300K tokens/day at no cost.
Zero downtime + independent verification = higher accuracy.
Why Redis?
Atomic writes prevent corrupted partial state on crash. The pipeline continues in degraded mode if Redis is unavailable — no data loss, just no crash recovery.
Crash recovery. Graceful degradation.
Why React + shadcn/ui?
Zustand for simpler state management than Redux. shadcn/ui for accessible, composable components. Tailwind for rapid styling. Bun for fast installs.
Full UI in ~2 weeks.
Why Azure (Key Vault + Container Apps)?
Key Vault: secrets rotate in one place, referenced at runtime via managed identity — no credentials in code or CI logs. Container Apps: serverless containers, scale-to-zero, built-in HTTPS.
Zero stored credentials. OIDC deploy. Auto-scaling.
Why Docker + GitHub Actions?
Multi-stage Dockerfile separates build (gcc/torch) from runtime (slim). 6-job CI pipeline: lint, test, security scan, Docker build, Azure deploy, benchmark. GHCR stores images.
Push to main → production in <10 minutes.

8-Node LangGraph Pipeline

Each node is a composite agent delegating to 1–4 specialists internally. Click a node for details.

Each SAS file flows through eight sequential stages. First, the file is scanned for cross-file dependencies and registered. A streaming finite-state machine then tokenises the SAS source, and a boundary detector splits it into logical partitions. The RAPTOR module clusters these partitions into a hierarchical embedding tree for richer retrieval. A machine-learning classifier assigns a risk level to each partition, determining which retrieval and translation strategy to apply. After persisting partitions and building a dependency graph, the core translation node converts each partition using three tiers of retrieval-augmented generation, with formal verification and adversarial testing. Finally, all translated partitions are merged into a single executable Python script with a quality report.

3-Tier RAG System

Retrieval-Augmented Generation adapted per partition complexity. The risk-routing node decides which tier each block receives.

Tier 1: Static RAG

When: LOW risk, no cross-file deps
How: KNN in LanceDB (k=5, cosine)
Speed: ~50ms

The partition source is embedded into a 768-dimensional vector. The five nearest knowledge-base examples are injected as context for translation.

~60% of partitions use this tier.

LOW RiskNo Dependencies

Tier 2: Graph RAG

When: Cross-file deps or SCC member
How: Dependency graph traversal + ancestor context
Speed: ~200ms

Traverses the dependency graph to include translations of upstream partitions as additional context.

~15% of partitions.

Cross-fileSCC Members

Tier 3: Agentic RAG

When: MOD/HIGH/UNCERTAIN risk
How: Multi-step retrieval (k=10), entity extraction
Speed: ~500ms

Extracts entities, queries the knowledge base from multiple angles, detects failure modes. Cross-verification uses an independent LLM provider.

~25% of partitions. +15% accuracy vs no RAG.

MOD/HIGH RiskMulti-step

The retrieval strategy is chosen automatically based on each partition's risk level and dependency profile. Simple, isolated blocks get fast nearest-neighbour search. Blocks with cross-file dependencies receive context from already-translated upstream partitions. Complex or high-risk blocks trigger a multi-step retrieval process with entity extraction, failure mode detection, and independent cross-verification by a second LLM provider.

RAPTOR: Hierarchical Embedding Tree

Recursive Abstractive Processing for Tree-Organised Retrieval. Click any node for details.

Interactive RAPTOR Tree

Instead of flat nearest-neighbour search, RAPTOR embeds each partition into a 768-dimensional vector space, then clusters them using Gaussian Mixture Models. Each cluster receives a summary that captures shared semantics across its members — for example, "running total followed by group aggregation." These summaries are re-embedded and re-clustered recursively, forming a tree whose root captures the full-file abstraction. When translating a complex partition, the system retrieves context at the appropriate tree level, capturing cross-partition relationships invisible to flat search.

Why Not Flat KNN?

Flat KNN misses cross-partition context
RAPTOR's cluster summaries capture semantic links like "running total followed by group aggregation" — invisible to flat KNN but critical for accurate translation.
+12% translation accuracy on MOD/HIGH blocks vs flat KNN.

Z3 Formal Verification

11 SMT patterns providing mathematical proofs of translation correctness — or concrete counterexamples showing exactly where the translation breaks.

Why Z3? Why not just run the code?
Running code tests ONE input. Z3 proves correctness for ALL possible inputs simultaneously. When a proof fails, it returns the exact breaking input as a counterexample.
Counterexample blocks are re-queued at HIGH risk for retry.
Z3 covers ~30%. What about the rest?
Z3 works on arithmetic and Boolean formulae. CDAIS covers the other 70% with adversarial DataFrames. Together: ~95% verification coverage.

For each translation, the system extracts decidable fragments — arithmetic expressions, Boolean filters, sort directions, join conditions — and encodes them as satisfiability problems. The Z3 solver then either proves that the Python output is semantically equivalent to the SAS original for all possible inputs, or returns a concrete counterexample showing the exact values that cause divergence. Blocks that fail verification are automatically re-queued with elevated risk for another translation attempt.

CDAIS: Adversarial Input Synthesis

For the 70% that Z3 cannot encode, CDAIS synthesises the minimum adversarial DataFrame that exposes translation errors.

CDAIS generates the smallest possible test DataFrame containing edge cases specific to each SAS pattern — missing values, currency strings, group boundaries, duplicate keys, mixed-case text, and numeric extremes. Each synthesised DataFrame is designed to trigger exactly the kind of divergence that would occur between SAS and Python when the translation is incorrect, catching bugs that random testing would miss.
