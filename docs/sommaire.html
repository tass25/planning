"This is the global architecture of Codara. My primary goal here was to design a system that is highly scalable and secure. To truly understand it, let me walk you through the exact journey a user takes when using the platform.

It all starts at the React frontend. A user arrives and securely logs in using either standard credentials or GitHub OAuth. Once authenticated, they upload their legacy SAS files into my drag-and-drop interface. The React app then securely transmits these files over HTTPS to my FastAPI backend, which acts as the central brain of the system.

Once the backend receives the files, it doesn't just translate them blindly. It triggers my core AI Translation Engine, which is built on an 8-node LangGraph pipeline. Instead of a simple linear script, this pipeline acts as an intelligent state machine. It routes the SAS files through different agents based on complexity, tests the Python output, and if an error is detected, it automatically triggers a fallback mechanism to retry or use a different model. During this entire process, the backend constantly communicates with my Data Layer—pulling exact syntax from my vector database and tracking the translation state in real-time.

But where does all of this live? I plan to deploy this decoupled system on Azure Container Apps. What is it? It is a serverless platform that automatically scales up when the platform has many users translating files simultaneously, and scales down to zero when idle, saving costs.

Finally, how do I keep this safe? I use Azure Managed Identity. This allows my backend to securely talk to my databases and my LLMs without ever storing a password or API key in the code. By removing hardcoded credentials, I completely eliminate the risk of a security leak."

"You might ask: why use four different databases? Let me explain the visual structure of each.

First, Redis. As you can see, it acts as a central hub of connected nodes. It is an in-memory database used because my AI pipeline executes asynchronously. Redis allows me to instantly 'save the state' (checkpointing) so the AI can pause and resume without losing its memory.

Second, LanceDB. Notice the clustered dots. LanceDB stores 'vectors' (mathematical representations of code). I chose it because it runs directly inside my application, avoiding slow network delays.

Third, SQLite. This is represented by a traditional table. It handles everyday data like user sessions safely and locally.

Finally, DuckDB. Notice the vertical columns. It is an analytical powerhouse that reads data column-by-column, allowing me to perform heavy audits on my translation metrics, analyzing millions of rows locally in milliseconds."

Slide 5: The LangGraph Pipeline

5 / 11

🖼️ What to put on the slide

A diagram of the 8-node LangGraph pipeline (extract this from the middle of your main architecture diagram).

"At the heart of my backend is the translation engine. I built this using a framework called LangGraph, which allows me to build robust 'state-machines' instead of simple linear scripts. Let me walk you through the 8 sequential nodes of this machine:

1. File Processing: Because enterprise SAS projects rarely exist in isolation, this node resolves all cross-file dependencies and traces data lineage before parsing even begins.

2. Streaming: SAS syntax is irregular, so I built a finite-state machine to reliably parse the source code at high speed—processing 10,000 lines in under 2 seconds.

3. Chunking: Massive files cannot be translated all at once due to LLM limits. This node intelligently splits the token stream into logical partitions using grammar-based rules and LLM fallback.

4. RAPTOR: Flat vector search misses global context. I implemented Gaussian Mixture Models to recursively cluster and summarize partitions. This allows the AI to grasp the 'big picture', boosting accuracy by 12% on complex blocks.

5. Risk Routing: Using a 14-feature machine-learning model, this node predicts the complexity of each chunk. Simple code gets fast processing, while difficult code is routed to heavy, agentic retrieval.

6. Persist & Index: This node acts as a save point. It writes to SQLite for crash recovery and builds a dependency graph so we translate chunks in the exact right order.

7. Translation (The Core): This is the heart of the system. It uses an LLM fallback chain across multiple providers to guarantee uptime. Here, code is translated, then immediately sent to the Verification Layer for mathematical proofs and adversarial testing. If it fails, the node automatically retries.

8. Merge: Finally, all successfully translated chunks are stitched back together into a single, clean Python script with deduplicated imports and an HTML quality report."

Slide 6: RAG vs Graph RAG vs Agentic RAG

6 / 11

🖼️ What to put on the slide

Use this connected infographic to explain the three types of RAG used in the pipeline.

RAG

Standard RAG

How we built it: LanceDB vector search on raw code chunks.
When to use it: Retrieving exact syntax mappings (e.g., SAS `proc sql` to Python `pandas`).

GRAPH

Graph RAG (RAPTOR)

How we built it: Gaussian Mixture Models clustering into a hierarchical tree.
When to use it: Retrieving global context and understanding how files interact.

AGENT

Agentic RAG

How we built it: LangGraph nodes that dynamically query databases based on failures.
When to use it: When the AI realizes it lacks context and actively searches for missing imports.

"To make my LLM truly intelligent, I don't just use one retrieval method. I use an evolving strategy.

First, what is Standard RAG? It is a simple vector search. I built it using LanceDB to retrieve exact syntax mappings. I use it for simple, isolated code chunks.

However, standard RAG loses the big picture on large codebases. So I introduced Graph RAG (using RAPTOR). How did I build it? I use clustering algorithms to group related code and summarize them into a hierarchy tree. When do I use it? When the AI needs to understand the global architecture of the project, not just a single function.

Finally, I use Agentic RAG. What is it? Unlike normal RAG which searches passively, Agentic RAG gives the AI the ability to think. I built it into my LangGraph nodes. When the AI fails a compilation test because of a missing import, the Agent actively decides to query the database again to find the exact file it needs to fix the error."

"A major challenge with Generative AI is hallucination. In the financial or defense sectors, I cannot simply trust an AI to translate a critical formula blindly.

To solve this, my current ongoing research involves building a deterministic Verification Layer. This layer uses three distinct tools.

First, the Sandbox Exec. What is a Sandbox? It is an isolated, secure environment where I can safely run the Python code the AI just generated. Why use it? Because it instantly catches obvious syntax errors or infinite loops without endangering my main system.

But catching crashes isn't enough; I need to know the math is correct. This is where Z3 and CDAIS come in.
What is Z3? It is a mathematical theorem prover developed by Microsoft Research. Unlike normal testing, which requires test data, Z3 converts the original SAS code and the new Python code into mathematical equations. Why use it? Because it can mathematically prove that the logic is identical for literally every possible input, capturing edge cases that human tests miss.
Finally, what is CDAIS? It is an adversarial engine that intentionally generates tricky, complex data to try and break the translated code. If any of these three tools flag an error, the system automatically rejects the code and falls back to an alternative model."

Slide 8: DevOps & CI/CD Roadmap

8 / 11

🖼️ What to put on the slide

A simple flow chart infographic: GitHub -> GitHub Actions (Tests) -> Docker Build -> GHCR -> Azure Container Apps.

"An enterprise system is only as strong as its deployment pipeline.

As part of the immediate next steps in my roadmap, I am designing a strict DevOps pipeline.

What is CI/CD? It stands for Continuous Integration and Continuous Deployment. Whenever an engineer updates the translation logic, GitHub Actions will automatically run hundreds of legacy SAS tests. If they pass, the application is packaged into a Docker Container—which is essentially a standardized box containing all the code and dependencies.

This container is then pushed to the GitHub Container Registry (GHCR) and deployed safely to Azure. This automated flow guarantees that I never accidentally break the translation engine when adding new features."
