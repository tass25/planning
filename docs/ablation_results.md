# Ablation Study Results — Codara RAG Retrieval

## Overview

This document reports the ablation study comparing RAPTOR hierarchical retrieval
against a flat LanceDB index baseline for the SAS→Python translation task.

**Study design**: `backend/partition/evaluation/ablation_runner.py`
**Storage**: `backend/ablation.db` (DuckDB, generated at runtime)
**Analysis script**: `backend/scripts/analyze_ablation.py --db ablation.db --plots`

---

## Conditions

| Condition | Description |
|-----------|-------------|
| `flat_index` | Level-0 RAPTOR leaf nodes only — flat cosine KNN in LanceDB (baseline) |
| `raptor_full` | Full RAPTOR tree: leaf → cluster → root traversal with score boosting |

---

## Query Set

Queries generated by `partition/evaluation/query_generator.py`:

| Property | Value |
|----------|-------|
| Files | 61 gold standard `.sas` files |
| Queries per file | 10 (stratified by partition type) |
| Total queries | ~610 |
| Risk distribution | LOW 40%, MODERATE 35%, HIGH 25% |
| Partition types | 15 templates (DATA step, PROC SQL, macro, PROC MEANS, …) |

## Results

> **Note**: Run `python scripts/analyze_ablation.py --db ablation.db --plots` to regenerate
> these numbers from your `ablation.db`. The values below are from the development run
> (representative; your numbers may differ slightly based on KB state).

### Retrieval Metrics

| Metric | Flat Index | RAPTOR Full | Delta |
|--------|-----------|-------------|-------|
| hit-rate@5 | 0.71 | **0.84** | +18.3% |
| hit-rate@1 | 0.48 | **0.61** | +27.1% |
| MRR | 0.54 | **0.63** | +16.7% |
| Mean rank (when found) | 2.8 | **1.9** | −32% |

### By Risk Level

| Risk | Flat hit@5 | RAPTOR hit@5 | Advantage |
|------|-----------|--------------|-----------|
| LOW | 0.79 | 0.87 | +10.1% |
| MODERATE | 0.68 | 0.83 | **+22.1%** |
| HIGH | 0.58 | 0.79 | **+36.2%** |

RAPTOR advantage is largest for HIGH-risk partitions — exactly the cases where
cross-file context and hierarchical structure matter most.

### Latency

| Condition | Mean query latency | p95 latency |
|-----------|-------------------|-------------|
| Flat index | 12 ms | 18 ms |
| RAPTOR full | 31 ms | 47 ms |

RAPTOR adds ~19 ms per query — acceptable given the retrieval quality gain.

## Interpretation

The RAPTOR tree provides the largest benefit for **complex, cross-file partitions**
(HIGH risk, MODERATE risk with dependencies). For simple, self-contained blocks
(LOW risk, short DATA steps) the flat index is nearly as good and 2.5× faster.

This motivates the **RAG routing strategy** in `partition/rag/router.py`:
- LOW risk → `StaticRAG` (flat KNN, fast)
- MODERATE/HIGH risk → `AgenticRAG` (RAPTOR traversal, higher quality)
