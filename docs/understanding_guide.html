Understanding Your Project from Zero

A plain-language guide to Codara, CDAIS, MIS, Z3, and everything in the research paper — explained with examples, not jargon.

🗺️ The Big Picture — What Did You Build?

You built two things. Don't confuse them:

🏗️ Codara

A software system — a full web app with a backend pipeline, frontend dashboard, database, and API. It takes SAS files and translates them to Python automatically.

📄 The Research Paper

A scientific contribution — two new methods (CDAIS + MIS) that check whether the translated Python is actually correct. Submitted to SCAM 2026 conference.

🎯 One-sentence summary of the whole project: Codara automatically converts legacy SAS code to Python, and uses formal mathematical methods (Z3, CDAIS, MIS) to prove the translation is correct — not just check that it runs.

The problem with existing tools: they convert the code and say "done." You run the Python and it works... but produces wrong numbers silently. Your contribution is catching those silent bugs.

🐛 The Problem — Why Is This Hard?

SAS and Python think differently

SAS was built in the 1970s with a specific way of processing data row by row. Python's pandas library processes entire columns at once. This difference creates bugs that are very hard to see.

The classic example — the RETAIN bug

Imagine you have sales data for two regions (A and B), and you want to compute a running total that resets at each region.

SAS code:

SAS — What you want
data output;
  set sales;
  by region;
  retain total 0;
  if first.region then total = 0;
  total + amount;
run;
Python — What LLM often writes (WRONG)
df['total'] = df['amount'].cumsum()
# No group reset!
# .cumsum() keeps adding forever

Now look at what happens with the data:

A3060
← resets here for new region
B55

B1530
❌ WRONG output (buggy Python)
RegionAmounttotal
A1010

← NO reset, keeps counting!
B565

⚠️ The dangerous part: The buggy Python code runs without any error. It produces a DataFrame. It looks reasonable. If you only have Region A in your test data, you will never catch this bug. The wrong numbers go into a financial report, a clinical trial, a regulatory filing — and nobody knows.

This is the exact problem your paper solves. You need to detect bugs that pass execution tests but produce wrong output.

🏗️ The Codara System — How Does It Work?

Codara is an 8-stage pipeline. Think of it like a factory assembly line where each station does one job:

1
File Process Reads SAS files, finds which files depend on which others (like finding imports in Python)
2
Streaming Parser Reads the SAS file token by token using a state machine — never loads the whole file into memory
3
Chunking Cuts the SAS code into logical blocks (DATA step, PROC SORT, macro, etc.) — each block translates independently
4
RAPTOR Clustering Groups similar code blocks together so when translating, we can retrieve similar examples from the knowledge base
5
Risk Routing Scores each block: LOW / MODERATE / HIGH complexity. Simple blocks get fast translation, complex ones get more resources
6
Persist & Index Saves everything to the database, builds a dependency graph so complex blocks can reference each other
7
Translation The main step: retrieves similar examples from the knowledge base (RAG), calls the LLM to translate, then validates using Z3, CDAIS, and MIS
8
Merge Combines all translated blocks into one Python script, fixes imports, resolves dependencies, generates a report

The 3-Tier RAG (Knowledge Retrieval)

When translating a block, the system doesn't just ask the LLM blindly. It first searches a knowledge base of 330 verified SAS→Python examples to give the LLM relevant context:

🔵 Static RAG (LOW risk) Simple KNN search — find 3 most similar examples
🟡 Graph RAG (MODERATE) Follows dependency links — if block uses a macro from another file, retrieves that file's context too
🔴 Agentic RAG (HIGH) Multi-step retrieval — searches multiple times, refines queries, escalates to more examples

🔢 What is Z3? (The Math Engine)

Think of Z3 like a super-powered Sudoku solver. You give it a set of mathematical rules and constraints, and it either finds values that satisfy all the rules, or proves that no such values exist.

Z3 is a tool made by Microsoft Research called an SMT Solver (Satisfiability Modulo Theories). You write mathematical formulas, and Z3 answers: "Is there a concrete example that satisfies this?"

Simple example of what Z3 does:

Z3 Python usage — toy example
from z3 import *
x, y = Ints('x y')
solver = Solver()
solver.add(x + y == 10)
solver.add(x > 3)
solver.add(y > 3)
solver.check()  # → SAT (satisfiable)
solver.model()  # → x=4, y=6  (a concrete answer!)

How Codara uses Z3 — two ways:

Way 1: Formal Verification (proving correctness)

For simple patterns like "this block just sorts data" or "this block filters rows by a condition", Z3 can mathematically prove that the Python translation produces the same result as SAS for all possible inputs.

Result: UNSAT on the "are these different?" query = they are never different = translation is correct.

This works for 30% of blocks. The other 70% are too complex for Z3 to prove directly.

Way 2: CDAIS — Finding the minimum test data (the paper's main contribution)

For bugs that Z3 can't prove away, we use Z3 differently: instead of proving correctness, we ask Z3 to find the smallest possible dataset that will expose the bug.

This is the CDAIS method — explained in full in the next section.

🎯 CDAIS — The Core Method of Your Paper

CDAIS = Constraint-Driven Adversarial Input Synthesis

Long name, simple idea. Let me break it down word by word:

🔒
Constraint-Driven

The test data is generated from mathematical rules (Z3 constraints), not randomly guessed

⚔️
Adversarial

The data is specifically designed to expose bugs — it's the worst-case input for a given error type

🧬
Input Synthesis

We synthesize (mathematically construct) the test data — we don't sample or guess

The CDAIS process — step by step

1

Look at the SAS code — what bug class could be here?

Does this block use RETAIN + BY + FIRST? → could have RETAIN_RESET bug. Does it use MERGE without IN=? → could have JOIN_TYPE bug. etc.

2

Write Z3 constraints describing when the bug would be visible

For RETAIN_RESET: "I need data with at least 2 groups, and the values in group 1 must be non-zero, because that's when cumsum() and groupby().cumsum() give different results."

3

Ask Z3 to find the SMALLEST values satisfying those constraints

Z3 returns: 2 groups, 3 rows each, all values = 1. That's the minimum dataset that exposes the bug.

4

Run both SAS oracle and Python translation on this 6-row dataset

If outputs differ → bug confirmed. If outputs match → issue a Coverage Certificate.

The actual RETAIN_RESET witness (6 rows)

Z3 gives us this exact data:

🔬 CDAIS Witness — RETAIN_RESET (6 rows, generated in 31ms)
groupvaluecorrect_total (SAS)buggy_total (Python .cumsum())
A111

↑ group A fine ↓ group B: the bug appears here
B114 ← WRONG

🔑 Why this is better than random testing: Random testing has only a ~72.5% chance of exposing this bug in each attempt (you might accidentally generate single-group data). CDAIS has a 100% guarantee in 1 attempt — because Z3 proved this data structure exposes the bug.

The Coverage Certificate — what it means

If the Python translation passes the CDAIS witness test (produces the same output as the SAS oracle), we issue a Coverage Certificate: "This translation is free from RETAIN_RESET bugs for any dataset with the same structure (2 groups, 3 rows each)."

This is not just "it passed a test" — it's a mathematical proof backed by Theorem 1 in the paper.

The SORT_STABLE special case (17 rows)

For SORT_STABLE bugs (unstable sort), the minimal witness needs 17 rows instead of 6. Why? Because Python's numpy uses insertion sort (stable) for arrays of ≤16 elements, and only switches to quicksort (unstable) at n=17. A 2-row or 6-row test would never trigger the instability. Z3 synthesis takes 296ms for this one (vs 31ms for others).

⚠️ The 6 Error Classes — Every One Explained

These are the 6 types of bugs your system is designed to detect. Each one is a pattern where SAS and Python behave differently:

C1

RETAIN_RESET

Running total that should reset per group but doesn't

RETAIN total 0; IF FIRST.grp THEN total=0;
df['total'] = df['val'].cumsum() ❌
df.groupby('grp')['val'].cumsum() ✓
C2

LAG_QUEUE

SAS LAG() maintains queue across whole dataset, not per group

lag_val = LAG(x); /* resets at BY boundaries */
df['lag_val'] = df['x'].shift(1) ❌
groupby().shift(1) per group ✓
C3

SORT_STABLE

SAS PROC SORT is always stable; Python default sort is not

PROC SORT DATA=ds; BY key; RUN;
df.sort_values('key') ❌ (quicksort)
df.sort_values('key', kind='mergesort') ✓
C4

NULL_ARITHMETIC

SAS SUM() skips missing values; Python x+5 propagates NaN

z = SUM(x, 5); /* returns 5 if x is missing */
df['z'] = df['x'] + 5 ❌ (NaN + 5 = NaN)
df['x'].fillna(0) + 5 ✓
C5

JOIN_TYPE

SAS MERGE without IN= is an outer join; Python defaults to inner

DATA out; MERGE left right; BY key; RUN;
pd.merge(left, right, on='key') ❌ (inner!)
pd.merge(..., how='outer') ✓
C6

GROUP_BOUNDARY

FIRST./LAST. selects first row per group, not first row of whole DataFrame

IF FIRST.x THEN output;
df.head(1) ❌ (whole DF, not per group)
df.groupby('x').first() ✓
✅ Remember: All 6 bugs have one thing in common — the Python code runs without error. That's what makes them dangerous. Your paper is the first to use formal SMT synthesis to detect all 6 with a mathematical guarantee.

📐 MIS — Migration Invariant Selection

MIS = Migration Invariant Selection

CDAIS targets known bug types. MIS handles a different question: "What properties should ANY correct SAS→Python translation always have?"

Imagine you've collected 142 blood samples from perfectly healthy people. You notice that ALL of them have a white blood cell count between 4,000–11,000. So you write the rule: "if a sample has WBC outside this range, something is wrong." You didn't invent this rule from theory — you discovered it from data. That's MIS.

How MIS works

1

Start with 18 candidate rules (analyst-written)

Example rules: "the output should never be empty if the input isn't", "after a GROUP BY, the output should have fewer rows than the input", "column data types should stay numeric"

2

Collect 142 verified correct (SAS, Python) pairs

12 pairs validated by two independent LLMs + manual review. 130 enterprise pairs validated by a senior SAS developer at Deloitte.

3

Test each rule against every applicable pair

Run the SAS oracle and check if the oracle output satisfies the rule. If the oracle violates it even once → the rule is wrong about SAS → REJECT. If oracle always satisfies it → CONFIRM.

4

10 rules confirmed, 8 rejected

The 10 confirmed rules are now a reusable checklist. Any new translation that violates one of these has a semantic bug — guaranteed.

Examples of confirmed vs rejected invariants

InvariantStatusWhy
OUTPUT_NONEMPTY✅ ConfirmedIf input is non-empty, output must be non-empty (universal)
COLUMN_DTYPE_STABILITY✅ ConfirmedNumeric columns stay numeric — always true in correct translations
RETAIN_MONOTONE_CUMSUM✅ ConfirmedRETAIN accumulators always increase — true in the corpus
MERGE_OUTER_ROWCOUNT✅ ConfirmedOuter merge output ≥ max(left, right) rows
COLUMN_SUPERSET❌ Rejected96.6% oracle pass — SAS KEEP/DROP can remove input columns → rule too strict
SORT_KEY_SORTED❌ Rejected8.3% oracle pass — NODUPKEY removes rows in ways that break simple sort order
LAG_NULL_FIRST_ROW❌ Rejected0% oracle pass — revealed a bug in our own LAG oracle! Fixed immediately.
💡 Key insight about the 8 rejections: These aren't failures — they're discoveries. SORT_KEY_SORTED seemed obviously true ("PROC SORT sorts, right?") but failed because SAS NODUPKEY changes row counts in complex ways. LAG_NULL_FIRST_ROW failing at 0% told us our own oracle had a bug in it, which we then fixed. MIS is a self-correction tool.

Leave-One-Out Cross-Validation (LOO-CV)

To make sure the 10 confirmed invariants are stable (not just lucky on this specific dataset), we did LOO-CV: we removed each of the 142 pairs one at a time and re-ran MIS on the remaining 141. Result: zero invariants changed their decision across all 142 folds. The confirmed set is stable.

🏛️ The 5-Layer Validation Pipeline

After the LLM translates a SAS block, the translation passes through 5 validation layers. Each catches different bugs:

1
Sandbox Execution Just runs the Python code and checks: does it crash? Returns True/False. Catches syntax errors and crashes. Does NOT check if output is correct.
2
Z3 Formal Verification For simple patterns (sort, filter, simple arithmetic): mathematically proves the translation is correct for ALL inputs. Works on 30% of blocks. If it returns UNSAT → formally correct.
3
Semantic Validator / Oracle Runs both the SAS oracle (a Python re-implementation of SAS semantics) AND the translation on the same synthetic data. Compares outputs: same rows? Same values? Same column types?
4
CDAIS Adversarial Testing Synthesizes Z3 witnesses for each applicable error class (C1–C6). If the translation fails → CEGAR repair loop: gives the bug to the LLM to fix, then re-verifies.
5
MIS Invariant Checking Runs all 10 confirmed invariants against the translation output. A violation means the translation breaks a property that ALL 142 verified correct translations satisfy.
✅ Why all 5 layers? Layer 1 only checks that code runs. Layer 2 only covers 30% of blocks. Layer 3 covers more but uses a Python oracle (imperfect). Layer 4 covers the 6 known bug classes. Layer 5 catches unknown bugs via invariant violation. Each layer catches things the others miss.

All 4 verification signals (Z3 proof, oracle diff, CDAIS certificates, MIS violations) are combined into one score: the Semantic Correctness Score (SCS).

SCS LevelMeaningWhat it takes
VERIFIEDFormally correctZ3 proof + all layers pass
LIKELY_CORRECTStrong evidence of correctness≥3 layers pass
UNCERTAINMixed signals2 layers pass
LIKELY_INCORRECTSemantic bug detected<2 layers pass

The missed targets — be honest about these

Boundary Detection: 79.3% instead of 90%

Why it missed: The lark grammar (the parsing tool) doesn't expand macros before detecting boundaries. A SAS macro that expands to 200 lines looks like just 1 line to the detector. Macro-heavy code gets only 52.1% F1.

Fix needed: Build a macro pre-processor that expands macros before boundary detection.

Hard-Tier Translation: 53.7%

Why it missed: Same root cause — macros, hash objects, deeply nested constructs. The 53.7% is still better than any existing tool (which produce 0% on these files because they refuse to process them).

🎓 How to Explain This in Your Defense

If they ask "what is your main contribution?"

"My main scientific contribution is CDAIS: the first use of SMT synthesis to generate formally minimal test inputs for LLM-based code migration. For each of 6 formalized error classes, I encode the divergence condition as a Z3 constraint, synthesize the minimum dataset that exposes the bug in 1 trial, and issue a mathematically proved coverage certificate. This replaces random testing — which needs hundreds of trials with no guarantee — with a 73ms deterministic process."

If they ask "explain Z3 simply"

"Z3 is a constraint solver — you give it mathematical rules and it finds concrete values that satisfy them, or proves no such values exist. I use it two ways: to prove simple translations are correct for all inputs, and to synthesize the minimum dataset that exposes a bug in complex translations."

If they ask "why did you miss the 90% boundary target?"

"The boundary detector uses a partial SAS grammar. It correctly identifies DATA steps and PROC steps (91% F1) but fails on macro-parameterised code (52% F1) because macros expand to arbitrary code at runtime — you can't detect boundaries in macro calls without first expanding them. The fix is a macro pre-processor, which is the highest-priority future work."

If they ask "what is a coverage certificate?"

"A coverage certificate is a formal guarantee, proved by Theorem 1, that if a translation passes the CDAIS witness test, it is free from that specific error class for any dataset with the same structural shape. It is not 'the test passed' — it is 'mathematical proof that the bug cannot exist for data of this structure'."
