SAS → Python / PySpark · Global Architecture

How it works — end to end
  1. A single .sas file—or an entire project of hundreds—enters through Project Entry. Each file is received, decoded (auto-detect encoding via chardet), and forwarded to the Entry layer.
  2. The File Registry (L2-A) catalogs every discovered file, computes a SHA-256 content hash to avoid processing duplicates, resolves cross-file references (%INCLUDE, LIBNAME paths), and extracts table-level data lineage—which datasets each block reads from and writes to—persisting everything into SQLite.
  3. Registered files flow into the Streaming Core. The File Stream Reader opens each file asynchronously (8 KB buffered reads) and feeds tokens line-by-line to the Streaming State Manager, a finite-state machine that tracks nesting depth, macro scope, and block boundaries in real time. At the same time, the Complexity Analyzer scores each emerging block’s logical and structural complexity using cyclomatic-complexity metrics and five statistical features.
  4. The stream feeds into the Partition Detector (L2-C), powered by the RAPTOR algorithm. A Lark LALR grammar identifies block boundaries (80 % rule-based); the remaining 20 % of ambiguous cases are resolved by an LLM. Detected blocks are then clustered with Gaussian Mixture Models (BIC-based k-selection, τ = 0.72), building a recursive semantic tree. The Partition Builder assembles each cluster into one of 9 canonical partition types (DATA_STEP, PROC_BLOCK, MACRO_DEFINITION, SQL_BLOCK…).
  5. In parallel, the complexity metadata flows down to the Strategy Selector (Control & Strategy Decision Layer). Based on the risk level of each partition, it decides the downstream strategy: which LLM to use, whether RAPTOR clustering is needed, and how much human review is required—all governed by a Platt-calibrated confidence model (ECE < 0.08).
  6. Partitions, their RAPTOR tree, and associated metadata are persisted into the Partition & KB Store (L2-E): SQLite for structured data, a Kuzu graph for dependency edges (with SCC condensation for circular %INCLUDE chains), and LanceDB for vector embeddings used by downstream retrieval.
  7. Stored partitions flow into Layer 3 — Translation & Validation. The Translation Agent queries the Knowledge Base for the 5 most similar SAS→Python/PySpark example pairs (filtered by failure-mode), routes the block to an 8B model (low risk) or 70B model (moderate/high risk) for code generation, and cross-verifies the result. The Validation Agent then executes the generated code on a synthetic 100-row DataFrame in a sandboxed namespace (5 s timeout) to confirm correctness.
  8. Validated translations reach Layer 4 — Merge & Report. The Script Merger consolidates imports, injects cross-file variable dependencies, and gates every output through ast.parse(). The Report Agent generates a full Markdown + HTML conversion report with CodeBLEU scores, validation results, and failure-mode breakdowns.
  9. The final output—converted .py / _spark.py scripts plus conversion reports—is written to disk and logged in DuckDB for auditability.
  10. The Continuous Learning loop closes the circle. Corrections (human CLI or automated rejects) are cross-verified and upserted into the Knowledge Base. The Quality Monitor tracks success rates and failure-mode distributions; when ECE drifts above threshold or quality drops, it triggers automatic retraining of the complexity model and targeted KB generation for under-represented categories.
