Phase-by-Phase Walkthrough — with examples

Persistence Layer (Shared across all phases — Durable State & Recovery)

Before diving into the phases, it’s important to understand that every phase reads from and writes to a shared persistence layer. This means the system can crash at any point and resume without re-processing. Four storage engines work together:

EngineWhat it storesWhy
SQLiteFile registry, partition metadata, complexity scores, strategy decisionsFast structured queries, single-file portability
NetworkXFile → file dependencies, table → table lineage, %INCLUDE chainsIn-memory graph: traversal, SCC detection, topological sort
LanceDBVector embeddings of every partition + KB examplesSemantic similarity search for translation retrieval
DuckDBAudit logs, conversion metrics, quality historyAnalytical queries on historical runs

Phase 1 · INGEST (Organize the project)

What happens?

The user uploads one or more .sas files (or an entire project folder). Two components handle this phase:

Component A — Project Entry

Receives the upload, auto-detects each file’s encoding using chardet (handles EBCDIC, Latin-1, UTF-8, etc.), decodes the raw bytes into clean text, and forwards every file to the File Registry.

Component B — File Registry (L2-A)

The registry does five things:

  • Catalog — persists file name, path, size, and hierarchy into SQLite
  • SHA-256 dedup — computes a content hash so identical files are never processed twice
  • Resolve %INCLUDE — follows %include "utils.sas"; to find referenced files
  • Resolve LIBNAME — maps library references like libname mylib "/data"; to actual paths
  • Extract data lineage — identifies which datasets each block reads from (SET, MERGE, FROM) and writes to (DATA, CREATE TABLE)
✓ Output of Phase 1 — File Manifest (Clean Project Map)
Files: - main.sas (hash: A91F…, encoding: UTF-8) - utils.sas (hash: B72D…, encoding: Latin-1) Dependencies (%INCLUDE / LIBNAME): main.sas → includes → utils.sas main.sas → libname → mylib (/data/) Data Lineage (SET/MERGE/FROM → DATA/CREATE): raw.sales → sales_clean → summary

So now the system understands: what files exist, how they depend on each other, what tables are created and what tables are read. It builds a dependency graph:

raw.sales ↓ sales_clean ↓ summary

All of the above is persisted into SQLite + NetworkX graph. If the system restarts, it resumes from here.

Phase 2 · ANALYZE (Understand the code structure)

The File Manifest from Phase 1 now enters the Streaming Core, which has three tightly coupled sub-components that work together in real time.

Sub-component 1 — File Stream Reader (L2-B)

Opens every registered file asynchronously using aiofiles with an 8 KB buffer. It reads the file line by line and emits raw tokens into a stream. This means even very large SAS files (50,000+ lines) are processed without loading the entire file into memory.

Its output: a raw token stream — one token per line.

Sub-component 2 — Streaming State Manager (L2-B)

This is a Finite-State Machine (FSM) that consumes the raw token stream and annotates every single line with its parsing context. It tracks:

  • Are we inside a DATA step? A PROC? A MACRO?
  • What is the current nesting depth? (e.g., a macro inside a macro)
  • What is the current macro scope? (local vs. global variables)
  • Where are the block boundaries? (start/end of each logical unit)
Example — Before vs. After Annotation

Instead of just seeing data sales_clean;, the State Manager produces:

Line 3 Type: DATA_STEP_START Block: DATA_STEP Depth: 1 Macro Scope: GLOBAL Line 9 Type: PROC_SQL_START Block: SQL_BLOCK Depth: 1 Macro Scope: GLOBAL

Its output: an annotated token stream where every line carries its full parsing context.

Sub-component 3 — Complexity Analyzer (L2-D)

Consumes the annotated stream and scores each code block using radon (cyclomatic complexity) plus 5 statistical features:

#FeatureWhat it measures
1Cyclomatic ComplexityNumber of independent paths through the code
2Nesting DepthMax depth of nested IF/DO/SELECT blocks
3Line CountTotal lines in the block
4Macro InvocationsNumber of macro calls (%macro_name)
5SQL ComplexityJOIN count, subquery depth, CASE expressions

Combined, these produce a risk score for each block:

BlockTypeCyclomatic ComplexityNestingLinesRisk
sales_cleanDATA_STEP214Low
summary SQLSQL_BLOCK4212Medium
format_dateMACRO103Low
✓ Output of Phase 2 — Each block now has a complexity record
Block ID: 1 Type: DATA_STEP Complexity Score: 2 Nesting Depth: 1 Line Count: 4 Risk Level: Low

These complexity records flow into both Phase 3 (for partitioning) and the Strategy Selector (for LLM routing decisions).

Phase 3 · PARTITION (Break into intelligent chunks)

The annotated stream + complexity scores from Phase 2 now enter the RAPTOR partitioning engine, which has three sub-components working in sequence.

Sub-component 1 — Partition Detector (L2-C · RAPTOR)

Detects block boundaries using a two-tier approach:

  • 80% rule-based — A Lark LALR grammar parses SAS syntax and identifies where blocks start and end (data...run;, proc...quit;, %macro...%mend;)
  • 20% LLM fallback — For ambiguous or malformed code that the grammar can’t parse, an LLM resolves block boundaries

Once boundaries are found, blocks are clustered using Gaussian Mixture Models (GMM):

  • BIC (Bayesian Information Criterion) automatically selects the number of clusters (k)
  • Cluster threshold τ = 0.72 controls how aggressively blocks are grouped
  • The result is a recursive semantic tree of related code blocks
Sub-component 2 — Partition Builder (L2-C)

Takes each cluster and assigns it to one of 9 canonical partition types:

#Partition TypeExample SAS Pattern
1DATA_STEPdata ... ; set ... ; run;
2PROC_BLOCKproc sort / proc freq / proc means ...
3MACRO_DEFINITION%macro name; ... %mend;
4SQL_BLOCKproc sql; ... quit;
5MACRO_INVOCATION%my_macro(param=value);
6MACRO_INCLUDE%include "file.sas";
7LIBNAME_STATEMENTlibname lib "path";
8GLOBAL_STATEMENToptions ...; title ...; footnote ...;
9COMMENT_BLOCK/* multi-line comment */
Partition Output Example
Partition 1: Type: MACRO_INCLUDE Source: %include "utils.sas"; Risk: Low Partition 2: Type: DATA_STEP Code: data sales_clean; set raw.sales; revenue = price * quantity; run; Risk: Low Partition 3: Type: SQL_BLOCK Code: proc sql; create table summary as select region, sum(revenue) as total_rev from sales_clean group by region; quit; Risk: Medium
RAPTOR Clustering — Semantic Tree

Related blocks are clustered into a hierarchical tree:

Project ├── Macros │ └── format_date (MACRO_DEFINITION) ├── Data Processing │ └── sales_clean (DATA_STEP) └── Aggregation └── summary (SQL_BLOCK)
Sub-component 3 — Strategy Selector (L2-D · Control & Decision Layer)

Runs in parallel with partitioning. It receives the complexity metadata and makes three decisions for each partition:

  • Which LLM? — 8B model for low risk, 70B model for medium/high risk
  • RAPTOR depth? — whether deeper recursive clustering is needed
  • Human review? — whether a human must validate the translation

All decisions are governed by a Platt-calibrated confidence model with ECE < 0.08 (Expected Calibration Error), ensuring probability estimates are trustworthy.

Partition 2 → Risk: Low → Use 8B model → No human review Partition 3 → Risk: Medium → Use 70B model → Auto-review only
Storage — Partition & KB Store (L2-E)

Everything from Phase 3 is persisted into three engines:

  • SQLite → partition metadata, types, risk levels, strategy decisions
  • NetworkX → dependency edges between partitions, SCC condensation for circular %INCLUDE chains (if A includes B and B includes A, they become one node)
  • LanceDB → vector embeddings of each partition for semantic retrieval in Phase 4

Phase 4 · TRANSLATE (Convert SAS → Python / PySpark)

Stored partitions from Phase 3, along with their strategy decisions, now enter Layer 3 — Translation & Validation. This layer has two agents.

Agent 1 — Translation Agent (L3)

The Translation Agent selects one of three RAG paradigms based on the partition’s risk level and dependency profile:

① Static RAG — for LOW risk, no cross-file dependencies

  • Fixed k = 3 retrieval from LanceDB (leaf level only)
  • Single-pass generation via the 8B model (Ollama)
  • No cross-verification, no retrieval retry — fastest path

② GraphRAG — for blocks with cross-file dependencies or SCC membership

  • Before KB retrieval, traverses the NetworkX dependency graph (nx.ancestors()) to collect upstream translations already produced
  • SCC siblings (circular dependency groups) are gathered as additional context
  • k = 5 retrieval at cluster level (related procedure families)
  • Generation via the 70B model (Groq) with cross-verification enabled
  • Generation retry on failure (not retrieval retry)

③ Agentic RAG — for MODERATE / HIGH risk blocks or detected failure modes

  • The agent autonomously decides whether to retrieve (UNCERTAIN blocks → skip retrieval, emit stub)
  • Adaptive k: starts at 5, escalates to 8, then 11 on failure
  • On cross-verification failure: query reformulation (appends failure mode + error to query) + retrieval retry (not just generation retry)
  • RAPTOR level escalation: leaf → cluster → root
  • Final escalation: root-level retrieval combined with graph context from nx.ancestors()
  • If all retries fail: block marked PARTIAL for human review

Selection logic: Has cross-file deps or SCC? → GraphRAG. No deps + LOW risk? → Static RAG. No deps + MOD/HIGH risk? → Agentic RAG. Deps + MOD/HIGH risk? → GraphRAG + Agentic combined.

Example — DATA_STEP Translation:
SAS Input
data sales_clean; set raw.sales; revenue = price * quantity; run;
Python (pandas) Output
sales_clean = raw_sales.copy() sales_clean["revenue"] = sales_clean["price"] * sales_clean["quantity"]
PySpark Output
from pyspark.sql import functions as F sales_clean = raw_sales.withColumn( "revenue", F.col("price") * F.col("quantity") )
Example — SQL_BLOCK Translation:
SAS Input
proc sql; create table summary as select region, sum(revenue) as total_rev from sales_clean group by region; quit;
Python (pandas) Output
summary = ( sales_clean .groupby("region", as_index=False) .agg(total_rev=("revenue", "sum")) )
PySpark Output
summary = ( sales_clean .groupBy("region") .agg(F.sum("revenue").alias("total_rev")) )
Agent 2 — Validation Agent (L3)

Immediately after generation, the Validation Agent runs every translated block through an automated test:

  • Creates a synthetic 100-row DataFrame matching the expected schema
  • Executes the code inside a sandboxed namespace using exec()
  • Enforces a 5-second timeout (kills infinite loops)
  • Checks: columns exist? · types valid? · no runtime errors? · date ranges valid?

If it passes → approved and moves to Phase 5.
If it fails → retried (up to 3 attempts with adjusted prompts), then flagged for human review if still failing.

✓ Output of Phase 4 — Validated Python + PySpark Blocks
✔ DATA_STEP → main.py translated & validated ✔ DATA_STEP → main_spark.py translated & validated ✔ SQL_BLOCK → main.py translated & validated ✔ SQL_BLOCK → main_spark.py translated & validated

Phase 5 · ASSEMBLE (Build final script & report)

Validated translations from Phase 4 now enter Layer 4 — Merge & Report, which has two agents.

Agent 1 — Script Merger (L4)

Takes every validated block and assembles them into complete scripts. It does three things:

  • Import consolidation — collects all import statements, removes duplicates, places them at the top
  • Dependency injection — if main.sas uses a variable from utils.sas, it ensures the variable is available in the merged script
  • ast.parse() syntax gate — every output script is parsed by Python’s ast module to guarantee it’s syntactically valid. If ast.parse() fails, the script is rejected and re-routed
Final Merged Script — main.py
import pandas as pd # --- From utils.sas --- def format_date(col): return pd.to_datetime(col, format="%Y%m%d") # --- From main.sas: DATA_STEP --- sales_clean = raw_sales.copy() sales_clean["revenue"] = sales_clean["price"] * sales_clean["quantity"] # --- From main.sas: SQL_BLOCK --- summary = ( sales_clean .groupby("region", as_index=False) .agg(total_rev=("revenue", "sum")) )
Agent 2 — Report Agent (L4)

Generates a detailed conversion report in both Markdown and HTML formats. The report includes:

  • CodeBLEU score — measures translation quality (0.0 to 1.0)
  • Per-block validation results — pass/fail for every partition
  • Failure-mode breakdown — categorizes any issues (e.g., “missing macro expansion”, “date format mismatch”)
  • Complexity summary — risk distribution across all blocks
  • Dependency graph visualization — which files/tables connect

Phase 6 · OUTPUT (Final deliverables)

The merged scripts and reports are written to disk. The complete output folder looks like:

output/ ├── main.py # Pandas version ├── main_spark.py # PySpark version ├── conversion_report.html # Visual report ├── conversion_report.md # Markdown report └── audit/ └── run_2026-02-23.duckdb # Full audit log
Report Summary Example
Project: sales_pipeline Files Processed: 2 (main.sas, utils.sas) Total Blocks: 3 Success Rate: 100% CodeBLEU: 0.91 Validation Failures: 0 Human Review Needed: 0
DuckDB Audit Log — what it records

Every single translation decision is logged, making the entire pipeline fully auditable.

Continuous Learning (System Gets Smarter)

The pipeline doesn’t just translate — it learns from every correction. Two components handle this.

Component 1 — Feedback Ingestion

Corrections can arrive in two ways:

  • Human CLI — a developer manually fixes a translation and submits the correction
  • Automated rejects — the Validation Agent flags a failed translation

Every correction is cross-verified (the fixed version is re-run through the Validation Agent to confirm it actually works), then upserted into the Knowledge Base as a new SAS → Python example pair.

Example — A Human Correction

The system originally generated:

sales_clean["revenue"] = sales_clean["price"] * sales_clean["quantity"]

A developer changes it to handle nulls:

sales_clean["revenue"] = ( sales_clean["price"].fillna(0) * sales_clean["quantity"].fillna(0) )

The system:

  • Detects the correction
  • Re-validates it (runs on synthetic DataFrame → passes)
  • Tags it with failure-mode: null_handling
  • Stores the better pattern in LanceDB as a new KB example
  • Next time a similar DATA_STEP appears, this example is retrieved and the LLM learns from it
Component 2 — Quality Monitor

Continuously tracks metrics across all runs:

MetricWhat it tracksTrigger
success_rate% of blocks translated without failureDrops below 85% → alert
failure_mode_distDistribution of failure categoriesAny category > 20% → targeted KB generation
ECE driftExpected Calibration Error of the complexity modelECE > 0.08 → automatic retraining
CodeBLEU trendAverage translation quality over timeDeclining trend → investigate

When the Quality Monitor detects a problem, it triggers:

  • Automatic retraining of the complexity model (re-calibrates risk predictions)
  • Targeted KB generation for under-represented failure categories (e.g., if “nested macros” keeps failing, the system generates more training examples for that pattern)
↺ The Feedback Loop
Correction detected ↓ Cross-verify (Validation Agent) ↓ Upsert into Knowledge Base (LanceDB) ↓ Quality Monitor updates metrics (DuckDB) ↓ If ECE drifts → retrain complexity model ↓ Next translation benefits automatically
Detailed Architecture Below
How it works — end to end
  1. A single .sas file—or an entire project of hundreds—enters through Project Entry. Each file is received, decoded (auto-detect encoding via chardet), and forwarded to the Entry layer.
  2. The File Registry (L2-A) catalogs every discovered file, computes a SHA-256 content hash to avoid processing duplicates, resolves cross-file references (%INCLUDE, LIBNAME paths), and extracts table-level data lineage—which datasets each block reads from and writes to—persisting everything into SQLite.
  3. Registered files flow into the Streaming Core. The File Stream Reader opens each file asynchronously (8 KB buffered reads) and feeds tokens line-by-line to the Streaming State Manager, a finite-state machine that tracks nesting depth, macro scope, and block boundaries in real time. At the same time, the Complexity Analyzer scores each emerging block’s logical and structural complexity using cyclomatic-complexity metrics and five statistical features.
  4. The stream feeds into the Partition Detector (L2-C), powered by the RAPTOR algorithm. A Lark LALR grammar identifies block boundaries (80 % rule-based); the remaining 20 % of ambiguous cases are resolved by an LLM. Detected blocks are then clustered with Gaussian Mixture Models (BIC-based k-selection, τ = 0.72), building a recursive semantic tree. The Partition Builder assembles each cluster into one of 9 canonical partition types (DATA_STEP, PROC_BLOCK, MACRO_DEFINITION, SQL_BLOCK…).
  5. In parallel, the complexity metadata flows down to the Strategy Selector (Control & Strategy Decision Layer). Based on the risk level of each partition, it decides the downstream strategy: which LLM to use, whether RAPTOR clustering is needed, and how much human review is required—all governed by a Platt-calibrated confidence model (ECE < 0.08).
  6. Partitions, their RAPTOR tree, and associated metadata are persisted into the Partition & KB Store (L2-E): SQLite for structured data, a NetworkX graph for dependency edges (with SCC condensation for circular %INCLUDE chains), and LanceDB for vector embeddings used by downstream retrieval.
  7. Stored partitions flow into Layer 3 — Translation & Validation. The Translation Agent queries the Knowledge Base for the 5 most similar SAS→Python/PySpark example pairs (filtered by failure-mode), routes the block to an 8B model (low risk) or 70B model (moderate/high risk) for code generation, and cross-verifies the result. The Validation Agent then executes the generated code on a synthetic 100-row DataFrame in a sandboxed namespace (5 s timeout) to confirm correctness.
  8. Validated translations reach Layer 4 — Merge & Report. The Script Merger consolidates imports, injects cross-file variable dependencies, and gates every output through ast.parse(). The Report Agent generates a full Markdown + HTML conversion report with CodeBLEU scores, validation results, and failure-mode breakdowns.
  9. The final output—converted .py / _spark.py scripts plus conversion reports—is written to disk and logged in DuckDB for auditability.
  10. The Continuous Learning loop closes the circle. Corrections (human CLI or automated rejects) are cross-verified and upserted into the Knowledge Base. The Quality Monitor tracks success rates and failure-mode distributions; when ECE drifts above threshold or quality drops, it triggers automatic retraining of the complexity model and targeted KB generation for under-represented categories.

Narrative Summary
The Journey of a SAS File — told as a story

Everything begins when a user drops one or more .sas files into the system. Phase 1 · Ingest The Project Entry component receives the upload, auto-detects each file’s encoding via chardet, and hands the raw source to the File Registry. The registry has four jobs: catalog the file in an SQLite index, compute a SHA-256 hash so duplicates are never processed twice, resolve cross-file references like %INCLUDE and LIBNAME paths, and extract table-level data lineage—which datasets each block reads from and writes to. Its output is a clean, deduplicated file manifest with a full dependency graph.

Phase 2 · Analyze That manifest enters the Streaming Core, which is made up of three tightly coupled sub-components. First, the File Stream Reader opens every registered file asynchronously with 8 KB buffered reads and emits tokens line by line. Those tokens flow into the Streaming State Manager, a finite-state machine that tracks nesting depth, macro scope, and block boundaries in real time—its output is an annotated token stream where every line carries its parsing context. In parallel, the Complexity Analyzer consumes that same annotated stream and scores each emerging code block on logical and structural complexity using cyclomatic-complexity metrics plus five statistical features. The result is a complexity metadata record attached to every block.

Phase 3 · Partition The annotated stream and its complexity scores now enter the Partition Detector, powered by the RAPTOR algorithm. A Lark LALR grammar identifies block boundaries (roughly 80 % rule-based); the remaining ambiguous 20 % are resolved by an LLM fallback. The detected blocks are then clustered using Gaussian Mixture Models (BIC-based k-selection, τ = 0.72), forming a recursive semantic tree. Those clusters are handed to the Partition Builder, which assembles each one into one of nine canonical partition types—DATA_STEP, PROC_BLOCK, MACRO_DEFINITION, SQL_BLOCK, and so on. Strategy Meanwhile, the complexity metadata also flows into the Strategy Selector: based on each partition’s risk level, it decides which LLM to use downstream, whether deeper RAPTOR clustering is needed, and how much human review is required—all governed by a Platt-calibrated confidence model (ECE < 0.08). Store Finally, partitions, their RAPTOR tree, strategy decisions, and all metadata are persisted into the Partition & KB Store: SQLite for structured data, a NetworkX graph for dependency edges (with SCC condensation for circular include chains), and LanceDB for vector embeddings that downstream retrieval will query.

Phase 4 · Translate Stored partitions now enter the Translation Agent. For each block, it queries the Knowledge Base for the five most similar SAS→Python/PySpark example pairs (filtered by failure-mode), routes the block to an 8B model if the risk is low or a 70B model if moderate-to-high, generates the Python/PySpark code, and cross-verifies the result. The generated code is immediately passed to the Validation Agent, which executes it on a synthetic 100-row DataFrame inside a sandboxed namespace with a 5-second timeout, checking column names, data types, and date-range invariants. Only validated translations move forward.

Phase 5 · Assemble The Script Merger takes every validated translation and consolidates imports, injects cross-file variable dependencies, and gates every output through ast.parse() to guarantee syntactic correctness. Alongside it, the Report Agent generates a full Markdown + HTML conversion report containing CodeBLEU scores, per-block validation results, and failure-mode breakdowns.

Phase 6 · Output The final artefacts—converted .py and _spark.py scripts plus their conversion reports—are written to disk and logged in DuckDB for full auditability.

Continuous Learning The story doesn’t end there. Every correction—whether submitted by a human through the CLI or flagged automatically by the Validation Agent—is cross-verified and upserted into the Knowledge Base. The Quality Monitor continuously tracks success rates and failure-mode distributions; whenever ECE drifts above threshold or quality drops, it triggers automatic retraining of the complexity model and targeted KB generation for under-represented categories. This feedback loop ensures the system gets smarter with every file it converts.
